2. Sysmon Fundamentals and Configuration

What Sysmon does and why configuration matters

Sysmon is a Windows system service and driver that persists across reboots and logs rich security telemetry. It supplements standard Windows event logs with detailed information about process creation, network connections, module loads, registry changes, and more. In real environments this data is usually forwarded into a SIEM like Splunk, Sentinel, or Elastic.

Configuration is what turns Sysmon from noise into signal. This lab references two widely used configuration projects to show how enterprise defenders tune their environments.

SwiftOnSecurity config

  • Well commented and easy to read.
  • Focuses on smart exclusions to cut down noise.
  • Good baseline for many enterprise deployments.

Olaf Hartong modular config

  • Maps event IDs to MITRE ATT&CK techniques.
  • Highly modular and customizable.
  • Geared toward analysts already comfortable with Sysmon.

In the portfolio report, I explain which events were most valuable in this lab and how a tuned configuration helps separate everyday noise from real attacker behavior.

3. High Value Event IDs and Filtering

Process, network, registry, and file telemetry for hunting

The lab focuses on a core set of Sysmon event IDs that map directly to common attack techniques. These events are where most of the hunting and investigation takes place.

Event ID Meaning Use in Lab
1 Process creation Detects malware, LOLBins, and first stage payloads.
3 Network connection Identifies beacons, callbacks, and suspicious ports.
7 Image loaded Helps spot unusual DLL loads and injection patterns.
8 Create remote thread Signals memory injection inside other processes.
10 Process access Used in the lab to detect Mimikatz touching LSASS.
11 File creation Surfaces droppers, payload files, and suspicious tools.
12 / 13 / 14 Registry events Tracks persistence keys and malware configuration.
15 File stream created Used for alternate data streams abuse.
22 DNS query Helps identify suspicious domains and C2 infrastructure.

I used both Event Viewer filters and PowerShell with XPath queries to narrow down telemetry and answer the investigation questions. PowerShell is especially useful when hunting for specific ports or processes. 

# Example: hunt for Sysmon network events on port 4444
Get-WinEvent -Path "C:\Windows\System32\Winevt\Logs\Microsoft-Windows-Sysmon%4Operational.evtx" `
-FilterXPath "*[System[EventID=3]] and *[EventData[Data[@Name='DestinationPort']='4444']]"

Using Event Viewer and PowerShell to filter Sysmon events by event ID, port, and process.

4. Threat Hunting Techniques

Beaconing, credential theft, persistence, and evasion

The lab walks through several hunting patterns that analysts use in real environments. Each technique focuses on a small set of event IDs, fields, and behaviors to keep the workflow repeatable and practical.

Metasploit beacon detection

  • Start with Sysmon Event ID 3 to find unusual ports like 4444 or 5555.
  • Pivot into Event ID 1 to see which process owns the connection.
  • Trace the parent process to understand how the payload started.

Mimikatz detection

  • Focus on Event ID 10 for process access.
  • Look for any process that accesses lsass.exe without a good reason.
  • Correlate with Event ID 1 to identify the calling binary and its parent.

Persistence detection

  • Use registry events (12, 13, 14) to find Run keys and hidden payload storage.
  • Hunt for scheduled tasks that launch PowerShell with encoded commands.
  • Flag file writes to startup locations or roaming profiles as potential persistence.

Evasion and alternate data streams

  • Event ID 15 surfaces alternate data stream usage.
  • Event IDs 7 and 8 help catch code that injects into other processes.
  • Combined with file creation and network events, this shows how the attacker hides.

5. Practical Investigations

Four full investigations based on real attack chains

The core of the lab is a series of investigations where Sysmon is used to answer very specific incident response questions. Each investigation becomes a case in the final SOC style report with commands, event IDs, and clear conclusions.

Investigation 1 — Malicious USB device

Goal: understand what happened when a malicious USB device was inserted.

  • Used registry events to find keys under HKLM\SYSTEM\CurrentControlSet\Enum\USBSTOR.
  • Reviewed raw access read events to identify the volume name.
  • Correlated with Event ID 1 to see rundll32.exe executing the payload from the USB.

Investigation 2 — HTML file dropping a payload

Goal: identify a masquerading HTML or HTA file that delivers a payload.

  • Tracked file creation events in temporary internet folders.
  • Found an HTA that pretended to be a downloaded HTML update.
  • Used Event ID 1 to confirm Microsoft signed mshta.exe launched the payload.
  • Mapped destination IP and port from Event ID 3 to show possible command and control traffic.

Investigation 3 — Registry and scheduled task persistence

Goal: uncover long term persistence through registry keys and scheduled tasks.

  • Located a registry key under HKLM\Software\Microsoft\Network\Debug that held a base64 payload.
  • Pivoted to PowerShell command lines that queried the registry value and executed the content.
  • Identified scheduled tasks that launched encoded PowerShell at a fixed time each day.
  • Used Event ID 10 to show the scheduled task eventually accessed lsass.exe, hinting at credential theft.

Investigation 4 — Botnet and command and control activity

Goal: confirm command and control behavior and infrastructure.

  • Used Event ID 3 to tie victim IP to an adversary controlled address and port.
  • Confirmed that the traffic matched a known Empire framework pattern.
  • Linked the network events back to the original process creation and parent chain.

6. SOC Style Report and MITRE ATT&CK Mapping

From raw telemetry to an analyst ready incident report

The final phase of the lab turns all investigations into a structured incident report similar to what a SOC analyst would write. It includes an executive summary for leadership and deep technical details for security teams.

Report structure

  • Executive summary with the overall story of the attacks and detections.
  • Environment section describing the Windows host, Sysmon setup, and logging paths.
  • Sysmon event breakdown for core event IDs used in the investigation.
  • Individual cases for each attack with commands, event IDs, indicators, and analysis.

MITRE ATT&CK mapping

  • Encoded PowerShell mapped to T1059.001 with ScriptBlock and Sysmon process events.
  • Run key persistence mapped to T1547.001 using registry modification events.
  • certutil or other LOLBins downloading payloads mapped to T1105 with process and network events.
  • Identity discovery commands like whoami mapped to T1033 using process creation events.
Simulation Technique Sysmon / Log Source Description
Encoded PowerShell T1059.001 Event ID 1 + ScriptBlock (4104) PowerShell command line and script content recorded.
Run key persistence T1547.001 Event ID 13 Registry value set to launch a payload at startup.
certutil style download T1105 Event ID 1 and 3 LOLBin used to fetch a remote payload over the network.
Account discovery T1033 Event ID 1 whoami and related commands used for local account discovery.

An optional advanced step is to forward Sysmon logs into a SIEM like Elastic, Splunk, or Microsoft Sentinel and build dashboards for process trees, command line usage, and LOLBin executions. The same data then supports more scalable threat hunting and alerting.

For an additional extension, a Linux system can be introduced as an external attacker. Simple actions like ping, curl, port scans, or brute force attempts generate traffic and events that Sysmon and Windows Defender will surface on the Windows host, creating a realistic attacker versus defender cycle for the portfolio.
