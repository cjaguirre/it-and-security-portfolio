The problem

The dataset was assembled from multiple public sources, making it noisy and inconsistent — duplicate rows, inconsistent company and industry labels, country names with trailing punctuation like United States., and dates stored as plain text strings that can't be sorted or aggregated reliably.

What I built

  • • Staging tables isolating all cleaning work from the raw source
  • • ROW_NUMBER() CTE to find and delete exact duplicates
  • • TRIM, LIKE, and UPDATE statements for text standardization
  • • Self-join to backfill missing industry values from same-company rows
  • • STR_TO_DATE() and ALTER TABLE to convert dates to native DATE type

Cleaning Walkthrough

Phases 1–2 — Import & Staging

The raw CSV is loaded into MySQL and profiled for issues. Then two staging tables that mirror the raw structure are created — all cleaning work happens in these copies so the original table is never touched. This makes the pipeline repeatable and safe to debug.

Staging setup code

-- Copy raw table into staging
CREATE TABLE layoffs_staging
  LIKE layoffs;

INSERT INTO layoffs_staging
SELECT * FROM layoffs;

-- Second staging layer for stepwise work
CREATE TABLE layoffs_staging2 (
  company              TEXT,
  location             TEXT,
  industry             TEXT,
  total_laid_off       INT DEFAULT NULL,
  percentage_laid_off  TEXT,
  `date`               TEXT,
  stage                TEXT,
  country              TEXT,
  funds_raised_millions INT DEFAULT NULL
);

INSERT INTO layoffs_staging2
SELECT * FROM layoffs_staging;

Phase 3 — Removing Duplicate Records

Exact duplicates occur when the same company, location, industry, layoff counts, and date appear more than once. A CTE with ROW_NUMBER() partitioned across all key fields flags these rows — then everything with a row number greater than 1 gets deleted.

Deduplication code

-- Flag duplicates with ROW_NUMBER()
WITH duplicate_cte AS (
  SELECT *,
    ROW_NUMBER() OVER (
      PARTITION BY
        company, location,
        industry, total_laid_off,
        `date`
    ) AS row_num
  FROM layoffs_staging
)
SELECT * FROM duplicate_cte
WHERE row_num > 1;

-- Delete the flagged duplicates
DELETE FROM layoffs_staging2
WHERE row_num > 1;

Why this approach works

  • CTE first, delete secondInspect exactly which rows will be removed before committing the delete — no surprises.
  • Partition by all key fieldsOnly rows identical across every meaningful column get flagged — partial matches are kept.
  • Staging protects sourceThe delete runs on staging2, not the original — the raw data is always recoverable.

Phase 4 — Standardization & Null Handling

Consistent text values are what make aggregations trustworthy. This phase fixes whitespace and capitalization in company names, fragmented industry labels (multiple variants of "Crypto"), and trailing punctuation in country names. Then missing industry values are backfilled via self-join, and records with no layoff data at all are removed.

Standardization code

-- Trim whitespace from company names
UPDATE layoffs_staging2
SET company = TRIM(company);

-- Consolidate industry label variants
UPDATE layoffs_staging2
SET industry = 'Crypto'
WHERE industry LIKE 'Crypto%';

-- Remove trailing punctuation from country names
UPDATE layoffs_staging2
SET country = TRIM(TRAILING '.' FROM country)
WHERE country LIKE 'United States%';

-- Backfill missing industry from same company's other rows
UPDATE layoffs_staging2 t1
JOIN layoffs_staging2 t2
  ON t1.company = t2.company
 AND t1.location = t2.location
SET t1.industry = t2.industry
WHERE t1.industry IS NULL
  AND t2.industry IS NOT NULL;

-- Drop rows where both layoff metrics are NULL (unusable)
DELETE FROM layoffs_staging2
WHERE total_laid_off IS NULL
  AND percentage_laid_off IS NULL;

The self-join backfill recovers missing industry labels without manual lookup — it just finds another row from the same company that has the value filled in.

Phase 5 — Date Conversion & Final Output

Dates stored as text strings can't be sorted chronologically or used in time series queries. STR_TO_DATE() parses the text format into a proper MySQL DATE, then ALTER TABLE changes the column type. The row_num helper column is dropped, leaving a clean final table ready for any BI tool.

Final conversion code

-- Parse text dates into DATE format
UPDATE layoffs_staging2
SET `date` = STR_TO_DATE(
  `date`, '%m/%d/%Y'
);

-- Change column type to native DATE
ALTER TABLE layoffs_staging2
MODIFY COLUMN `date` DATE;

-- Drop the deduplication helper column
ALTER TABLE layoffs_staging2
DROP COLUMN row_num;

-- Verify the final cleaned table
SELECT * FROM layoffs_staging2;

What native DATE unlocks

  • • Time series queries — group by month, quarter, or year
  • • Chronological sorting that actually works
  • • Date range filtering with standard WHERE clauses
  • • Direct connection to Tableau, Power BI, or any BI tool

What I Learned

Key takeaways

  • • Staging layers are essential — cleaning is iterative and you need to reset without re-importing raw data
  • • ROW_NUMBER() is the cleanest way to handle exact duplicates in SQL — auditable and precise
  • • Text standardization before aggregation matters more than it looks — one typo in a country name silently splits the data
  • • The self-join backfill is a useful pattern for recovering missing values from the same entity's other records
  • • Dates stored as text are a common real-world problem — converting them early makes every downstream query simpler

What the clean data enables

  • • Industry trends — which sectors shed the most jobs and when
  • • Geographic analysis — country-level comparisons without name collision errors
  • • Time series — monthly and quarterly layoff trends from the converted DATE column
  • • Funding stage analysis — how capital access relates to layoff scale
  • • Trustworthy totals — duplicates and unusable rows are gone
