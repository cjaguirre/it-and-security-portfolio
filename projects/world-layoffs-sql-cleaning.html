I rebuilt the dataset into a structured staging and cleaning pipeline in MySQL. The goal was to turn that noisy raw feed into an analysis-ready table that supports time-series reporting and layoff insights by company, industry, and country.

Dataset Issues & Staging Setup

The incoming layoffs dataset tracked companies, locations, industries, counts, and dates for workforce reductions, but it contained a mix of structural and quality issues:

  • Duplicate entries for the same company / date / location combination.
  • Inconsistent casing and whitespace in company and industry fields.
  • Industry variations such as multiple versions of “Crypto” labels.
  • Country names with trailing punctuation and inconsistent formatting.
  • Date values stored as strings instead of proper date types.

To avoid damaging the original data, I first duplicated the raw table into staging, then created a second staging table for incremental cleaning: 

-- Duplicate raw table for staging
CREATE TABLE layoffs_staging LIKE layoffs;
INSERT INTO layoffs_staging
SELECT * FROM layoffs;

-- Second staging table for cleaning work
CREATE TABLE layoffs_staging2 LIKE layoffs_staging;
INSERT INTO layoffs_staging2
SELECT * FROM layoffs_staging;

SQL Cleaning & Standardization

With the staging environment in place, I focused on removing duplicates, cleaning text fields, and consolidating inconsistent categories into a standard form.

  • Used a ROW_NUMBER() window to identify true duplicates.
  • Trimmed extra whitespace from company and country fields.
  • Standardized industry labels such as “Crypto” variants.
  • Normalized country names with trailing punctuation.
-- Remove duplicate rows
WITH cte AS (
    SELECT  *,
            ROW_NUMBER() OVER (
                PARTITION BY company, location, industry, date
            ) AS rn
    FROM layoffs_staging2
)
DELETE
FROM layoffs_staging2
WHERE rn > 1;

-- Clean up company names
UPDATE layoffs_staging2
SET company = TRIM(company);

-- Standardize industry values (example: Crypto)
UPDATE layoffs_staging2
SET industry = 'Crypto'
WHERE industry LIKE 'Crypto%';

-- Normalize country names by removing trailing periods
UPDATE layoffs_staging2
SET country = TRIM(TRAILING '.' FROM country)
WHERE country LIKE '%.%';

Transformations for Analytics

After cleaning string fields and removing duplicates, I prepared the data for analysis by converting text dates into proper DATE types and adding derived fields such as year.

  • Converted date strings using STR_TO_DATE() where needed.
  • Added a numeric year column for time-series aggregation.
  • Structured the table so BI tools can filter by year, company, industry, and country.
-- Convert date string column into a real DATE type (if stored as text)
UPDATE layoffs_staging2
SET date = STR_TO_DATE(date, '%Y-%m-%d');

-- Add and populate a year column
ALTER TABLE layoffs_staging2
ADD COLUMN year INT;

UPDATE layoffs_staging2
SET year = EXTRACT(YEAR FROM date);

Insights Enabled

With the cleaned and standardized layoffs table in place, the dataset now supports accurate workforce analytics across multiple dimensions:

  • Industry-level layoff intensity and trend cycles over time.
  • Country-by-country comparisons of layoff activity.
  • Breakdowns by funding stage, company size, or region (where available).
  • Aggregations by year and quarter for reporting and dashboarding.

Testing & Validation Approach

  • Row counts before/after: compared raw, staging, and cleaned tables to confirm only intended duplicates were removed.
  • Spot checks by company: manually verified that company / date combinations stayed unique after deduplication.
  • Null and outlier scans: checked key columns for unexpected NULLs or invalid values introduced during cleaning.
  • Re-ran summary aggregations before and after cleaning to confirm that totals still matched source expectations where possible.
