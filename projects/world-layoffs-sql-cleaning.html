Project overview

The world layoffs dataset was compiled from public sources and news reports, which meant it arrived messy and unreliable. Records were duplicated, company and industry labels were inconsistent, country names had stray punctuation, and dates were stored as text instead of proper types.

I implemented a repeatable cleaning pipeline inside MySQL that preserves the original raw table, runs transformations in staging layers, and outputs a cleaned dataset that can be used confidently for industry trends, geographic comparisons, and funding-related layoff analysis.

Key deliverables

  • Raw CSV import and profiling workflow in MySQL Workbench.
  • Two-layer staging design to isolate cleaning from the reference source.
  • Duplicate removal using ROW_NUMBER() over key business fields.
  • Standardization of company, industry, and country text values.
  • Null handling strategy that fills useful gaps and removes unusable rows.
  • Conversion of text-based dates into a native DATE column.
  • Documentation of setup, usage, and future automation opportunities.

Lab breakdown

1. Raw dataset overview

World layoffs CSV · company, location, industry, layoff counts, dates, funding

The original dataset captured company-level layoff events with columns for location, industry, total employees laid off, percent laid off, layoff date, funding stage, country, and funds raised. It arrived as a CSV file with no constraints and several visible quality issues.

  • Duplicate rows for the same company, location, date, and layoff counts.
  • Inconsistent capitalization and whitespace across text fields.
  • Country names such as United States. with trailing punctuation.
  • Dates stored as text strings instead of native date types.

Preview of the world layoffs CSV as originally delivered. Columns include company, location, industry, layoff counts, date, stage, country, and funds raised.

2. Staging and cleaning pipeline setup

Raw table preserved · transformations executed in staging layers

To keep the original layoff records untouched, I created staging tables that mirror the raw structure. All cleaning logic runs inside these staging layers, making the pipeline repeatable and easy to debug without risking the source data. 

-- Duplicate raw table into a staging table
CREATE TABLE layoffs_staging LIKE layoffs;

INSERT INTO layoffs_staging
SELECT *
FROM layoffs;

-- Create a second staging table for stepwise cleaning
CREATE TABLE layoffs_staging2 (
  company               TEXT,
  location              TEXT,
  industry              TEXT,
  total_laid_off        INT DEFAULT NULL,
  percentage_laid_off   TEXT,
  `date`                TEXT,
  stage                 TEXT,
  country               TEXT,
  funds_raised_millions INT DEFAULT NULL
) ENGINE = InnoDB
  DEFAULT CHARSET = utf8mb4
  COLLATE = utf8mb4_0900_ai_ci;

INSERT INTO layoffs_staging2
SELECT *
FROM layoffs_staging;

View of layoffs_staging2 in MySQL Workbench after copying from the raw table. This is the workspace for all subsequent cleaning operations.

3. Removing duplicate records with window functions

ROW_NUMBER() based deduplication across key business fields

Exact duplicates occur when the same company, location, industry, layoff counts, date, stage, country, and funding appear more than once. I used a common table expression with ROW_NUMBER() to flag those cases and then removed all but a single instance. 

-- Identify duplicates using ROW_NUMBER() and partition by business keys
WITH duplicate_cte AS (
    SELECT *,
           ROW_NUMBER() OVER (
               PARTITION BY company,
                            location,
                            industry,
                            total_laid_off,
                            percentage_laid_off,
                            `date`,
                            stage,
                            country,
                            funds_raised_millions
           ) AS row_num
    FROM layoffs_staging
)
SELECT *
FROM duplicate_cte
WHERE row_num > 1;

-- Insert into layoffs_staging2 with row_num
INSERT INTO layoffs_staging2
SELECT *,
       ROW_NUMBER() OVER (
           PARTITION BY company,
                        location,
                        industry,
                        total_laid_off,
                        percentage_laid_off,
                        `date`,
                        stage,
                        country,
                        funds_raised_millions
       ) AS row_num
FROM layoffs_staging;

-- Remove duplicate rows where row_num > 1
DELETE
FROM layoffs_staging2
WHERE row_num > 1;

After this step, each layoff event appears once per unique company, location, date, and layoff combination, which stabilizes totals and averages for downstream analysis.

4. Standardizing text fields and resolving missing values

Trimmed company names, consolidated industry labels, normalized country text, filled gaps

Consistent text values are critical for accurate group-by queries. I standardized company names, merged variant industry labels for cryptocurrency firms, cleaned up country names, and then addressed missing industries and unusable rows. 

-- Standardize company names by trimming whitespace
UPDATE layoffs_staging2
SET company = TRIM(company);

-- Consolidate variations of industries into a single category (Crypto)
UPDATE layoffs_staging2
SET industry = 'Crypto'
WHERE industry LIKE 'Crypto%';

-- Remove inconsistent formatting in the country field
UPDATE layoffs_staging2
SET country = TRIM(TRAILING '.' FROM country)
WHERE country LIKE 'United States%';
-- Populate missing industry fields by self-joining on company and location
UPDATE layoffs_staging2 t1
JOIN layoffs_staging2 t2
  ON t1.company = t2.company
 AND t1.location = t2.location
SET t1.industry = t2.industry
WHERE t1.industry IS NULL
  AND t2.industry IS NOT NULL;

-- Remove rows with no layoff information at all
DELETE
FROM layoffs_staging2
WHERE total_laid_off IS NULL
  AND percentage_laid_off IS NULL;

These steps keep as many informative records as possible while dropping rows that cannot contribute to quantitative analysis.

5. Date conversion and final cleanup

Converted text dates to DATE and removed helper columns

To support time-based analysis, the date column was converted from text to a native DATE using STR_TO_DATE(). Finally, the temporary row_num helper column from deduplication was dropped so the table is clean for long-term use. 

-- Convert date strings into native DATE values
UPDATE layoffs_staging2
SET `date` = STR_TO_DATE(`date`, '%m/%d/%Y');

-- Change the column type to DATE
ALTER TABLE layoffs_staging2
MODIFY COLUMN `date` DATE;

-- Drop helper column now that deduplication is complete
ALTER TABLE layoffs_staging2
DROP COLUMN row_num;

-- Verify final cleaned table
SELECT *
FROM layoffs_staging2;

At this stage, layoffs_staging2 serves as the cleaned, analysis-ready table that can be queried directly or exported into BI tools such as Tableau or Power BI.
