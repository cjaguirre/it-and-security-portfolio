The problem

The original data environment mixed spreadsheets, ad hoc tables, and partially designed schemas. Client records appeared in several places, bid amounts didn't tie to labor and material costs, and basic profitability questions required manual reconciliation every time.

What I built

  • • Normalized SQL Server schema with primary keys, foreign keys, and business constraints
  • • ERD covering clients, projects, bids, labor, materials, tools, and production planning
  • • Scripted deployment so the database can be rebuilt from source control
  • • 30+ analytical queries for revenue, demand, margin, and client analysis
  • • Testing and validation confirming constraints block invalid states

Database Breakdown

ERD & Schema Implementation

The ERD was the contract between business rules and technical implementation. It captures the full lifecycle of a landscaping project — from first client contact and bid through labor assignments, material sourcing, and production scheduling. The schema was then scripted in SQL Server so it can be deployed and rebuilt from source control.

Normalized ERD — clients, bids, labor, materials, tools, suppliers, and production planning

Key design decisions

  • • Client → Project (one-to-many) — one client can have many projects
  • • Project → Bid (one-to-many) — multiple bid iterations, one approved version
  • • Labor and materials linked via bridge tables (many-to-many)
  • • Production planning tied to the approved bid only
  • • Supplier linked to materials, not directly to projects

Schema script (excerpt)

CREATE TABLE Client (
  ClientID   INT IDENTITY(1,1) PRIMARY KEY,
  ClientName VARCHAR(100) NOT NULL,
  Phone      VARCHAR(20),
  Email      VARCHAR(100),
  City       VARCHAR(50),
  StateCode  CHAR(2)
);

CREATE TABLE Project (
  ProjectID   INT IDENTITY(1,1) PRIMARY KEY,
  ClientID    INT NOT NULL,
  ProjectName VARCHAR(120) NOT NULL,
  StartDate   DATE,
  EndDate     DATE,
  Status      VARCHAR(30),
  CONSTRAINT FK_Project_Client
    FOREIGN KEY (ClientID)
    REFERENCES Client(ClientID)
);

Analytical Query Library

The normalized schema enables a library of 30+ queries that answer real business questions without manual reconciliation. These can run directly in SSMS or feed a BI dashboard. The two patterns below cover the most common use cases: service demand and project profitability.

What the queries cover

  • • Which services drive the most revenue and how demand shifts by season
  • • High-value clients ranked by repeat work, total revenue, and profit contribution
  • • Estimated vs realized margins by combining bids with actual labor and material costs
  • • Supplier and material cost trends that may be eroding profitability

Service demand & client profitability

-- Popular services by volume
SELECT     s.ServiceName,
           COUNT(*) AS ProjectCount
FROM       Project p
JOIN       ProjectService ps ON p.ProjectID = ps.ProjectID
JOIN       Service s         ON ps.ServiceID = s.ServiceID
GROUP BY   s.ServiceName
ORDER BY   ProjectCount DESC;

-- Client profitability summary
SELECT     c.ClientName,
           p.ProjectName,
           SUM(jm.Quantity * m.UnitCost)      AS MaterialCost,
           SUM(l.HoursWorked * l.HourlyRate)  AS LaborCost,
           p.BidAmount - (
             SUM(jm.Quantity * m.UnitCost) +
             SUM(l.HoursWorked * l.HourlyRate)
           )                                   AS EstimatedProfit
FROM       Project p
JOIN       Client c           ON p.ClientID = c.ClientID
JOIN       JobMaterials jm    ON p.ProjectID = jm.ProjectID
JOIN       Material m         ON jm.MaterialID = m.MaterialID
JOIN       LaborAssignment l  ON p.ProjectID = l.ProjectID
GROUP BY   c.ClientName, p.ProjectName, p.BidAmount
ORDER BY   EstimatedProfit DESC;

These patterns can power dashboards that guide pricing, staffing, and which services to grow or phase out.

Testing & Roadmap

Validation focused on proving that the schema behaves correctly under real-world conditions — not just that it stores data, but that it blocks invalid states and that query results reconcile back to source numbers.

Testing approach

  • Parameter variationQueries tested across multiple time windows, regions, and service filters to confirm consistent behavior.
  • Data consistency checksTotals reconciled between bids, labor, and materials — every dollar traces back to a project and client.
  • Constraint testingInserts, updates, and deletes exercised to confirm foreign keys and unique constraints block invalid states.
  • Query performanceExecution plans reviewed and indexes added where analytics felt slow.

Future enhancements

  • Real-time integrationConnect the schema to operational systems via ETL or change data capture for live reporting.
  • Automated reportingSchedule weekly margin and workload snapshots for owners and project managers.
  • Expanded metricsIncorporate satisfaction scores, rework rates, and seasonal patterns for deeper analysis.
  • Multi-developer supportExtend contribution guidelines and naming conventions for team collaboration.

What I Learned

Key takeaways

  • • Normalization to 3NF eliminates redundancy and creates a single source of truth — the queries become trivial once the data model is right
  • • Modeling costs at the material and labor level (not just bid totals) is what unlocks accurate profitability analysis
  • • ERD relationships that mirror real workflows make it far easier to extend the database later without breaking existing queries
  • • Scripted deployment turns the database into a versioned asset — not just a file that lives on one machine
  • • Constraint testing is as important as query testing — if invalid data can be inserted, the analytics will silently mislead
