Landscape Database Revitalization

A complete rebuild of a fragmented landscaping company database into a clean, normalized, analytics-ready SQL Server system. Standardized the schema, enforced relational integrity, and built 30+ SQL queries to uncover customer trends, service patterns, and project profitability.

Advanced Database Management · Spring 2023 Tools: SQL Server, SSMS, ERDPlus

Project Narrative

The original environment mixed spreadsheets, ad-hoc tables, and partially implemented schemas. Client information was repeated across multiple tables, bid amounts did not tie cleanly to labor and material costs, and basic questions like “Which services are most profitable?” required manual reconciliation.

I redesigned the system from the ground up using an ERD-first approach, then implemented a fully normalized SQL Server database. The new structure standardizes business entities, enforces relationships, and supports fast, reliable reporting for both day-to-day operations and higher-level strategy.

What I Built

  • Normalized SQL Server schema with primary / foreign keys and business rules encoded as constraints.
  • ERD modeling for clients, projects, bids, labor, materials, tools, and production planning.
  • Standardized previously fragmented operational data into clean relational tables.
  • 30+ analytical SQL queries targeting customer behavior, service demand, and margin analysis.
  • Deployment, testing, and usage documentation for technical and non-technical stakeholders.

ER Modeling & Database Design

Using ERDPlus, I modeled the end-to-end lifecycle of a landscaping project — from the initial client and bid through materials, labor assignments, tools, and production scheduling. The ERD below drove the entire physical design in SQL Server.

  • Entities: Client, Project, DesignBid, Material, Labor, Tools, ProductionPlan, ProjectTasks, Suppliers, and supporting link tables.
  • Attributes: bid amounts, labor rates, material units and costs, task hours, approval dates, and schedule information.
  • Relationships: client–project (1:M), project–bid (1:M), project–labor and project–materials (M:N via bridge tables), and production planning tied back to the approved bid.
  • Cardinalities were refined to match real business rules (for example, a bid can be rejected multiple times but only one version is ultimately approved).

SQL Schema Implementation

The logical model was implemented in SQL Server Management Studio using a scripted, repeatable deployment. The setup script creates the database, builds core tables, establishes keys, and prepares the environment for test data and analytics. 

-- Create database
IF DB_ID('NBDDB_CTF') IS NOT NULL
    DROP DATABASE NBDDB_CTF;
GO

CREATE DATABASE NBDDB_CTF;
GO

USE NBDDB_CTF;
GO

-- Client table
CREATE TABLE Client (
    ClientID   INT IDENTITY(1,1) PRIMARY KEY,
    ClientName VARCHAR(100) NOT NULL,
    Phone      VARCHAR(20),
    Email      VARCHAR(100),
    City       VARCHAR(50),
    StateCode  CHAR(2)
);

-- Project table
CREATE TABLE Project (
    ProjectID   INT IDENTITY(1,1) PRIMARY KEY,
    ClientID    INT NOT NULL,
    ProjectName VARCHAR(120) NOT NULL,
    StartDate   DATE,
    EndDate     DATE,
    Status      VARCHAR(30),
    CONSTRAINT FK_Project_Client
        FOREIGN KEY (ClientID) REFERENCES Client(ClientID)
);

Schema scripts initialize the database, enforce relational integrity, and provide a clean baseline for loading sample or production data.

Analytical Insight Queries

On top of the normalized schema, I developed more than thirty queries to support managers, estimators, and operations leads. These queries can be used directly in SSMS or scheduled as the back end for dashboards and recurring reports.

  • Identify which services drive the most revenue and how demand shifts across seasons.
  • Track high-value clients based on repeat work, total revenue, and profit contribution.
  • Compare estimated versus realized margins by combining bid amounts with labor and material costs.
  • Highlight suppliers or materials where cost inflation is eroding profitability.
-- Popular services by volume
SELECT      s.ServiceName,
            COUNT(*) AS ProjectCount
FROM        Project p
JOIN        ProjectService ps ON p.ProjectID = ps.ProjectID
JOIN        Service s         ON ps.ServiceID = s.ServiceID
GROUP BY    s.ServiceName
ORDER BY    ProjectCount DESC;

-- Client profitability summary
SELECT      c.ClientName,
            p.ProjectName,
            SUM(jm.Quantity * m.UnitCost)     AS MaterialCost,
            SUM(l.HoursWorked * l.HourlyRate) AS LaborCost,
            p.BidAmount - (
                SUM(jm.Quantity * m.UnitCost) +
                SUM(l.HoursWorked * l.HourlyRate)
            ) AS EstimatedProfit
FROM        Project p
JOIN        Client c           ON p.ClientID = c.ClientID
JOIN        JobMaterials jm    ON p.ProjectID = jm.ProjectID
JOIN        Material m         ON jm.MaterialID = m.MaterialID
JOIN        LaborAssignment l  ON p.ProjectID = l.ProjectID
GROUP BY    c.ClientName, p.ProjectName, p.BidAmount
ORDER BY    EstimatedProfit DESC;

These patterns can be surfaced in dashboards to guide pricing, staffing, and which services to promote.

Testing & Debugging Approach

  • Parameter variation: validated queries using different time windows, regions, and service filters to confirm they behaved correctly under multiple scenarios.
  • Data consistency checks: cross-checked totals between modules (bids, labor, materials) to ensure that every dollar in the system reconciled back to a project and client.
  • Constraint and referential tests: exercised inserts, updates, and deletes to confirm that foreign keys and unique constraints blocked invalid states.
  • Reviewed execution plans and added indexes where necessary to keep analytical queries responsive.
