Landscape Database Revitalization

A complete redesign of a fragmented landscaping services database. The project rebuilt the schema from the ground up — fixing inconsistencies, removing duplicated logic, enforcing relationships, and creating a clean, analytics-ready SQL system for billing, job tracking, and operations reporting.

Operational Database Schema Refactoring

Project Narrative

The original database suffered from inconsistent client names, duplicated job entries, unstandardized service descriptions, and a lack of relational integrity. This made billing, material tracking, and reporting both slow and error-prone.

The system was redesigned using proper normalization principles — ensuring a clean structure, accurate joins, and consistent data ready for analysis, automation, and future dashboards.

What I Built

  • Fully normalized SQL schema covering clients, services, jobs, invoices, and materials.
  • Standardized service categories and cleaned inconsistent text fields.
  • Implemented relational keys and referential constraints.
  • Identified and eliminated duplicate job records.
  • Created a scalable structure supporting analytics and future CRM integration.

Problems Identified

  • Client names stored in inconsistent formats and with frequent misspellings.
  • Service descriptions lacked standard categories, causing reporting issues.
  • No primary keys or foreign key constraints anywhere in the schema.
  • Duplicate job entries with conflicting billing amounts.
  • Invoices mixed with job data, breaking normalization rules.
  • Free-text notes and manual timestamps impossible to analyze reliably.

New Database Schema

The redesigned schema follows proper relational modeling, reducing duplication and supporting reliable joins across all operational data.

  • Clients — standardized business names, contacts, locations.
  • Services — normalized service categories with pricing models.
  • Jobs — service type, job date, assigned crew, materials used.
  • Invoices — linked to jobs through strict foreign keys.
  • Materials — mulch, soil, turf, maintenance materials, etc.
  • JobMaterials — junction table capturing quantity + cost per job.

Sample SQL Improvements

-- Standardizing service categories
UPDATE Services
SET ServiceType = 'Lawn Maintenance'
WHERE ServiceType IN ('Lawn Care', 'Grass Cut', 'Cutting Grass');

-- Creating relational structure
ALTER TABLE Jobs
ADD CONSTRAINT fk_client
FOREIGN KEY (ClientID) REFERENCES Clients(ClientID);

-- Identifying duplicate jobs
SELECT JobID, ClientID, ServiceID, JobDate, COUNT(*)
FROM Jobs
GROUP BY JobID, ClientID, ServiceID, JobDate
HAVING COUNT(*) > 1;
