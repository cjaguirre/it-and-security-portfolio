Azure Sentinel Threat Analysis

Designed and deployed a Windows honeypot in Azure to capture and analyze global attack traffic. Forwarded event logs into Microsoft Sentinel, enriched malicious IPs with threat intelligence, and visualized real time attacker activity through an interactive world map.

Realistic SOC workflow simulation Tools: Azure, Log Analytics, Sentinel, KQL

Project Narrative

I created a simple, real-world security investigation by setting up a Windows computer in the cloud and leaving its remote access port open on purpose. This acted as a “honeypot,” attracting automated attackers from around the world.

As the attacks came in, the system recorded every failed login attempt. I then added location information to each attacker’s IP address and turned all the activity into clear visual charts using Microsoft Sentinel.

This project shows what happens when a server is left unprotected on the internet. Scanners quickly find it, attempt to break in, and generate a steady stream of security alerts. The lab demonstrates how an analyst collects, enriches, and visualizes this data to understand what’s happening and respond effectively.

What I Built

  • Set up an Azure account and all required resources.
  • Deployed a Windows 10 honeypot that was fully open to the internet.
  • Configured the network rules to allow all traffic in.
  • Turned off the Windows firewall to make the system intentionally vulnerable.
  • Connected the VM to a Log Analytics Workspace to collect live security logs.
  • Added the VM into Microsoft Sentinel for monitoring and analysis.
  • Uploaded a GeoIP file to identify where attacks were coming from.
  • Used a KQL function to match attacker IPs with their location.
  • Built a global attack map showing over 4,600 enriched security events.

Lab breakdown

1. Azure Environment Preparation

Ensured full access to core cloud services required for the security lab.

I began by verifying that my Azure subscription had everything needed for the lab. This included the ability to deploy virtual machines, create Network Security Groups, set up a Log Analytics Workspace, and activate Microsoft Sentinel. Confirming these core services up front mirrors the first steps a junior SOC analyst would take when preparing to work in a real cloud environment.

Azure home overview with core services surfaced.

Subscription view confirming resource availability.

2. Honeypot VM and intentional exposure

Deployed a Windows 10 VM, opened RDP, relaxed security rules, and removed host protections.

The goal for this step was to safely recreate what happens when a server is misconfigured on the internet. I deployed a Windows 10 VM with a public IP, opened its inbound Network Security Group rules, and disabled the Windows firewall. This allowed real attackers and automated scanners to immediately find the system, attempt logins, and generate authentic brute force activity for analysis.

  • Created a Windows 10 VM with RDP enabled.
  • Set NSG rules to allow inbound traffic from anywhere.
  • Disabled all Windows firewall profiles.wf.msc.
  • Generated initial failed logons using a decoy user account.

NSG rule intentionally configured to allow all inbound traffic.

3. Log forwarding and Microsoft Sentinel integration

Forwarded Windows security logs into Log Analytics using the Azure Monitor Agent and connected them to Sentinel.

Once the honeypot began producing SecurityEvent logs, I connected the VM to Azure’s monitoring pipeline so the data could be collected, stored, and analyzed in Microsoft Sentinel. This setup allowed every failed login attempt to flow directly into Log Analytics, where Sentinel could query and visualize the activity in real time.

  • Created a Log Analytics Workspace as the central logging location.
  • Activated Microsoft Sentinel on that workspace.
  • Used the Windows Security Events via AMA connector to install the Azure Monitor Agent and enable log collection.
  • Confirmed successful ingestion by querying Event ID 4625 (failed logons).

AMA connector showing a healthy connection from VM to Log Analytics.

4. GeoIP watchlist and KQL enrichment

Imported a GeoIP dataset and attached location data to SecurityEvent logs using KQL.

To make the source of each attack easy to understand, I brought a GeoIP dataset into Sentinel and used KQL to add country and region details to every failed logon. This allowed each IP address to be mapped to a real location, turning raw log data into something visual and meaningful.

  • Uploaded a CSV listing IPv4 ranges and their associated countries and regions.
  • Created a Sentinel watchlist named geoip, keyed by the network range.
  • Enriched 4625 failed logon events by joining them with the GeoIP watchlist using ipv4_lookup.

GeoIP watchlist loaded into Sentinel with country and region fields.

5. Sentinel workbook attack map

Built a workbook to visualize attacker locations using enriched failed-logon events.

To bring all the enriched data together, I created a custom Sentinel workbook that plots attacker IPs on an interactive map. The workbook runs the GeoIP-enhanced KQL query, displays each event by latitude and longitude, and includes controls that let me explore different time ranges and attack patterns.

  • Started with a blank workbook and added a query visual.
  • Connected the visual to the GeoIP-enriched KQL query.
  • Mapped latitude and longitude to the Azure map component.
  • Added filters for time ranges and dynamic data exploration.

Sentinel workbook map showing global failed logon activity.

Sample KQL

Baseline failed logons 

SecurityEvent
| where EventID == 4625
| sort by TimeGenerated desc

GeoIP enriched 4625 events 

let GeoIPDB_FULL = _GetWatchlist("geoip");
let WindowsEvents =
    SecurityEvent
    | where EventID == 4625
    | sort by TimeGenerated desc
    | evaluate ipv4_lookup(GeoIPDB_FULL, IpAddress, network);

Architecture overview

Components

  • Public internet — Source of real brute-force activity.
  • Azure NSG (Open Inbound) — Intentionally exposed to allow traffic.
  • Windows Honeypot VM — Windows host generating SecurityEvent logs.
  • Azure Monitor Agent — Sends logs to the workspace.
  • Log Analytics Workspace — Stores and indexes events.
  • Microsoft Sentinel — Runs KQL queries and visualizations.

Data flow

  • Internet Scanners → VM → Failed RDP attempts.
  • VM → AMA → Log Analytics.
  • Log Analytics → Sentinel → Analysis.
  • GeoIP watchlist → KQL → Attacker locations added.
  • Sentinel workbook → Map Visualization.

Key insights

  • Open RDP services are found and attacked extremely quickly.
  • The majority of traffic comes from automated infrastructure, not individuals.
  • Adding GeoIP data shows immediate geographic trends behind the attacks.
  • Visualizing events helps explain patterns clearly without reading raw logs.

Summary

This project shows how real attack telemetry can be captured, enriched, and visualized using Azure’s security stack. By deploying an intentionally exposed Windows host, collecting its logs, and combining them with GeoIP data in Sentinel, I created a streamlined analysis workflow that highlights global brute-force activity in a simple, understandable format. The finished workbook makes it easy for both technical and non-technical audiences to see where attacks originate and how they unfold in a cloud environment.
