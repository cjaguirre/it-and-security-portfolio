Azure Sentinel Threat Analysis

Designed and deployed a Windows honeypot in Azure to capture and analyze global attack traffic. Forwarded event logs into Microsoft Sentinel, enriched malicious IPs with threat intelligence, and visualized real time attacker activity through an interactive world map.

Realistic SOC workflow simulation Tools: Azure, Log Analytics, Sentinel, KQL

Project Narrative

I created a simple, real-world security investigation by setting up a Windows computer in the cloud and leaving its remote access port open on purpose. This acted as a “honeypot,” attracting automated attackers from around the world.

As the attacks came in, the system recorded every failed login attempt. I then added location information to each attacker’s IP address and turned all the activity into clear visual charts using Microsoft Sentinel.

This project shows what happens when a server is left unprotected on the internet. Scanners quickly find it, attempt to break in, and generate a steady stream of security alerts. The lab demonstrates how an analyst collects, enriches, and visualizes this data to understand what’s happening and respond effectively.

What I Built

  • Set up an Azure account and all required resources.
  • Deployed a Windows 10 honeypot that was fully open to the internet.
  • Configured the network rules to allow all traffic in.
  • Turned off the Windows firewall to make the system intentionally vulnerable.
  • Connected the VM to a Log Analytics Workspace to collect live security logs.
  • Added the VM into Microsoft Sentinel for monitoring and analysis.
  • Uploaded a GeoIP file to identify where attacks were coming from.
  • Used a KQL function to match attacker IPs with their location.
  • Built a global attack map showing over 4,600 enriched security events.

Workflow overview

Technical Workflow Phases

  • Phase 1: AZURE SETUP & EXPOSURE — Created Azure resources and subscription. Setup requires **unrestricting NSG rules** and disabling the Windows Firewall for intentional vulnerability.
  • Phase 2: HONEYPOT DEPLOYMENT — Deployed a **Windows 10 VM** with exposed RDP/SMB ports. This configuration acts as the **honeypot endpoint**, attracting automated attackers from the public internet.
  • Phase 3: LOG INGESTION — **Security event logs** are collected and forwarded to the **Log Analytics Workspace (LAW)**. Logs are ingested using the **Azure Monitor Agent (AMA)** for centralized SIEM analysis.
  • Phase 4: GEOIP WATCHLIST ADDITION — Prepared data for enrichment by **uploading and importing a GeoIP dataset** as a Sentinel watchlist, providing IP-to-country/region mapping data.
  • Phase 5: KQL ENRICHMENT & FILTERING — **Kusto Query Language (KQL)** is used to execute an `ipv4_lookup` join. This action **attaches geographic data** to security events (e.g., Event ID 4625), filtering for relevant attacks.
  • Phase 6: ATTACK MAP & ACTION — Built a **Sentinel workbook** to visualize the enriched data as a global **Attack Map**. The resulting threat intelligence is delivered to the SOC Analyst for response.

2. Honeypot VM and intentional exposure

Deployed a Windows 10 VM, opened RDP, relaxed security rules, and removed host protections.

To simulate how quickly an exposed system can attract unwanted activity, I deployed a Windows 10 VM with a public IP and removed its typical safeguards. Opening RDP, loosening its Network Security Group rules, and disabling the Windows firewall allowed the VM to behave like a misconfigured asset on the open internet. This setup invited real scanning traffic and automated login attempts, creating an authentic stream of security events for later analysis in Sentinel.

  • Deployed a Windows 10 VM with RDP enabled.
  • Configured NSG rules to allow inbound traffic from any source.
  • Disabled all Windows firewall profiles (wf.msc).
  • Generated initial failed logons using a decoy user account.

The screenshot below reflects the relaxed NSG configuration used during this stage. With restrictive filters removed, the VM became highly visible to external scanners—precisely the environment needed to collect real-world brute force attempts and reconnaissance traffic.

3. Log forwarding and Microsoft Sentinel integration

Forwarded Windows security logs into Log Analytics using the Azure Monitor Agent and connected them to Sentinel.

After the honeypot VM began generating real SecurityEvent activity, the next step was connecting it to Azure’s monitoring pipeline. By linking the VM to a Log Analytics Workspace and enabling Microsoft Sentinel, every failed login attempt and security event could be captured, stored, and analyzed in one place. This integration is what allowed Sentinel to begin visualizing the brute-force traffic hitting the exposed VM in real time.

  • Configured a Log Analytics Workspace as the centralized collection point.
  • Enabled Microsoft Sentinel on that workspace.
  • Installed the Azure Monitor Agent using the Windows Security Events via AMA connector.
  • Verified ingestion by querying Event ID 4625 (failed logons).

The connector view below reflects the moment ingestion was confirmed. Once the AMA pipeline was active, the workspace immediately began capturing authentication failures from the honeypot—providing a live feed of attacker activity for Sentinel dashboards and KQL investigations.

AMA connector showing a healthy connection from VM to Log Analytics.

4. GeoIP watchlist and KQL enrichment

Imported a GeoIP dataset and attached location data to SecurityEvent logs using KQL.

After confirming that failed logons were flowing into Sentinel, the next step was making the activity easier to interpret. Raw IP addresses don’t offer much insight on their own, so I imported a GeoIP dataset and used KQL to enrich each event with its corresponding country and region. This transformation turned anonymous login attempts into a clear picture of where attackers were coming from, adding important geographic context to the investigation.

  • Uploaded a CSV containing IPv4 ranges with mapped country and region fields.
  • Created a Sentinel watchlist named geoip, keyed by the network range.
  • Joined the watchlist with Event ID 4625 entries using ipv4_lookup to enrich failed logons.

The view below shows the enriched dataset inside Sentinel. With geographic details now attached, the SecurityEvent logs provided a much clearer narrative of attacker behavior and origin—setting the foundation for visual dashboards and the attack-map overlay built later in the lab.

GeoIP watchlist loaded into Sentinel with country and region fields.

5. Sentinel workbook attack map

Built a workbook to visualize attacker locations using enriched failed-logon events.

With geographic enrichment complete, I used a Sentinel workbook to turn the failed-logon data into a visual map of attacker activity. By connecting the workbook to the GeoIP-enhanced KQL query, each event could be plotted by latitude and longitude, revealing clear clusters of activity and patterns in where the attacks originated. The workbook also included time filters and interactive controls, making it easy to explore how attack volume changed throughout the day.

  • Began with a blank Sentinel workbook and added a KQL query visual.
  • Linked the visual to the GeoIP-enriched SecurityEvent query.
  • Mapped latitude and longitude fields to the Azure Maps component.
  • Added filters for time ranges, event counts, and dynamic exploration.

The map below represents the final visualization. By placing each failed-logon attempt on the world map, the workbook provided a clear, real-time view of attacker distribution and helped highlight which regions were generating the most persistent brute-force attempts.

Sample KQL

These queries were used to analyze failed RDP logons during the lab. The first retrieves raw failed-logon events, and the second enriches them with GeoIP data so attacker locations can be mapped and visualized.

Baseline failed logons 

SecurityEvent
| where EventID == 4625
| sort by TimeGenerated desc

GeoIP enriched 4625 events 

let GeoIPDB_FULL = _GetWatchlist("geoip");
let WindowsEvents =
    SecurityEvent
    | where EventID == 4625
    | sort by TimeGenerated desc
    | evaluate ipv4_lookup(GeoIPDB_FULL, IpAddress, network);

Architecture overview

Components

  • Public Internet — Real attackers scanning and brute-forcing exposed services.
  • Azure NSG (Open Inbound) — Deliberately left wide-open to attract traffic.
  • Windows Honeypot VM — Generates SecurityEvent logs from failed RDP attempts.
  • Azure Monitor Agent — Forwards SecurityEvent logs to Log Analytics.
  • Log Analytics Workspace — Central storage and query engine for collected logs.
  • Microsoft Sentinel — Runs KQL, enriches data, and visualizes attacker behavior.

Data flow

  • Scanners → Honeypot VM → Failed RDP events generated.
  • Azure Monitor Agent → Log Analytics ingestion.
  • Log Analytics → Sentinel for parsing and investigation.
  • GeoIP watchlist → KQL lookup → Country & region enrichment.
  • Sentinel workbook → Global attack-map visualization.

Key insights

  • Exposed RDP services attract scanning and brute-force activity within minutes of going online.
  • Most attempts originate from automated infrastructure rather than human attackers.
  • GeoIP enrichment immediately reveals geographic patterns and regional concentrations of activity.
  • Mapping enriched events provides faster insight than reviewing raw logs, highlighting trends at a glance.

Summary

This project demonstrates how real attack telemetry can be captured, enriched, and visualized using Azure’s security stack. By deploying an intentionally exposed Windows host, collecting its logs, and combining them with GeoIP enrichment in Sentinel, I created a streamlined investigation workflow that surfaces global brute-force activity in a clear and accessible format. The final workbook makes it easy for both technical and non-technical audiences to understand where attacks originate and how they unfold in a cloud environment.
