Azure SOC Honeypot and Attack Map Lab

This lab walks through building a Windows honeypot in Azure, exposing it to the public internet, forwarding security events into a SIEM, enriching attacker data with GeoIP context, and visualizing global attack activity on a real time map inside Microsoft Sentinel.

Built as a realistic Security Operations workflow
Tools: Azure, Log Analytics, Sentinel, KQL
Project Narrative
End to end SOC style workflow

The objective of this lab was to simulate what a Security Operations Center analyst does daily: collect logs from endpoints, route them into a central SIEM, enrich the data with context, then query and visualize that activity to understand attacker behavior.

I began by deploying a Windows 10 honeypot virtual machine in Azure and deliberately weakening its security posture by opening inbound traffic through the Network Security Group and disabling the Windows firewall. This encouraged real brute force attempts from the public internet and produced authentic security telemetry.

Using the Azure Monitor Agent and a Log Analytics Workspace, I forwarded Windows SecurityEvent logs (with a focus on failed logon events, ID 4625) into Microsoft Sentinel. From there, I built Kusto Query Language queries, enriched attacker IP addresses with country data using a GeoIP watchlist, and surfaced the results in an interactive attack map Workbook.

What I Built
High level checklist
  • Configured an Azure subscription and portal environment.
  • Deployed a Windows 10 honeypot virtual machine with public exposure.
  • Opened inbound access in the Network Security Group to invite real attacks.
  • Disabled Windows firewall on the host to fully expose RDP.
  • Forwarded Windows SecurityEvent logs to a Log Analytics Workspace via the Azure Monitor Agent.
  • Onboarded the workspace into Microsoft Sentinel as the SIEM layer.
  • Imported a GeoIP CSV file into Sentinel as a watchlist for location enrichment.

Workflow Overview
Stage Description
Azure Setup I started by creating an Azure subscription, signing into the Azure portal, and confirming that I had access to the services required for this lab: Virtual Machines, Network Security Groups, Log Analytics Workspaces, and Microsoft Sentinel.
Honeypot Creation and Exposure I deployed a Windows 10 virtual machine with a public IP address and RDP access. On the associated Network Security Group, I created an inbound rule allowing any protocol from any source to any destination. Inside the guest operating system I turned off all Windows firewall profiles using the wf.msc management console. This produced a fully exposed honeypot that would quickly draw inbound probing from the internet.
Local Log Validation Before forwarding logs to the cloud, I validated that the honeypot was actually producing meaningful events. I attempted several invalid logons using a fake user such as employee and inspected the Security log in Event Viewer. I confirmed that Windows was recording Event ID 4625 entries for failed logons, including information such as username, source IP, and failure reason.
Log Forwarding and SIEM Integration I created a Log Analytics Workspace (LAW), then enabled Microsoft Sentinel using that workspace as its data store. On the honeypot VM I configured the Windows Security Events via AMA connector, which installed the Azure Monitor Agent extension and created a Data Collection Rule. After the connector was active, I queried the SecurityEvent table in both LAW and Sentinel to confirm that failed logon events from the honeypot were arriving in the cloud.
Log Enrichment with GeoIP Raw SecurityEvent data includes source IP addresses but no geographic context. To add this, I imported a GeoIP CSV file as a Sentinel watchlist named geoip. Using KQL and the ipv4_lookup function, I joined attacker IP addresses from Event ID 4625 entries with the watchlist, enriching each event with country and other location attributes.
Attack Map Visualization With enrichment in place, I built a Sentinel Workbook that runs a KQL query against the SecurityEvent table and plots the results on a world map. The Workbook is configured to update as new failed logons are ingested, giving a near real time visualization of where attacks against the honeypot are originating.

Part Two: Honeypot Virtual Machine and Exposure

With the subscription ready, I moved on to creating the actual honeypot. The goal here was not to harden the host but to safely expose it in a controlled environment so that real attackers would attempt to access it.

VM deployment
  • Searched for Virtual Machines in the Azure portal and created a Windows 10 VM.
  • Enabled a public IP and left RDP open so that remote logons were possible.
  • Selected an appropriate machine size based on lab constraints and cost.
Deliberate exposure
  • Configured the VM Network Security Group with an inbound rule that allowed any source, any port, and any protocol.
  • Logged into the VM and used wf.msc to turn off all Windows firewall profiles (domain, private, public).
  • Confirmed that RDP and other ports were reachable from the internet.

This stage created a realistic target that would rapidly begin receiving scans and brute force attempts, replicating common misconfigurations that SOC teams see in real environments.

Part Three: Local Log Inspection

Before building out the full logging pipeline, I validated that Windows was generating the right events on the host.

  • Performed several failed RDP logons using a decoy username such as employee.
  • Opened Event Viewer, navigated to Windows Logs then Security, and searched for Event ID 4625.
  • Confirmed that each failed logon produced a 4625 entry, including time stamp, user name, and IP address details.

This step established a baseline: I knew that as soon as I connected the machine to Azure Monitor and Sentinel, those same 4625 events would be available for analysis.

Part Four: Log Forwarding and SIEM Integration

After confirming local logging, I created the cloud side infrastructure to collect and analyze events.

Central log store
  • Created a Log Analytics Workspace (LAW) that would act as the central log repository.
  • Enabled Microsoft Sentinel using this workspace as the backing data store.
  • Used the Security blade in Sentinel to manage connectors and analytics.
Connector configuration
  • Configured the Windows Security Events via AMA connector.
  • Automatically deployed the Azure Monitor Agent extension to the honeypot VM.
  • Created a Data Collection Rule to route SecurityEvent logs from the VM into LAW.
Sample KQL to verify ingestion
SecurityEvent
| where EventID == 4625
| sort by TimeGenerated desc

Running this query confirmed that the failed logon events generated on the honeypot were now visible inside both Log Analytics and Sentinel, which is the core of any SIEM pipeline.

Part Five: Log Enrichment and GeoIP Watchlist

Raw security logs are useful, but they become much more powerful when enriched with external intelligence. In this case, I wanted to know where in the world my attackers were located.

Watchlist setup
  • Downloaded the geoip-summarized.csv dataset containing IP ranges and country information.
  • Opened Microsoft Sentinel and created a watchlist named geoip.
  • Set the source type to local file and used network as the search key for CIDR blocks.
  • Verified that the import completed with about fifty four thousand rows.
KQL enrichment
  • Joined the SecurityEvent table to the GeoIP watchlist using ipv4_lookup.
  • Enriched each Event ID 4625 entry with country and related GeoIP attributes.
  • Used these enriched fields later as the basis for map coordinates.
GeoIP enrichment query
let GeoIPDB_FULL = _GetWatchlist("geoip");
let WindowsEvents =
    SecurityEvent
    | where EventID == 4625
    | order by TimeGenerated desc
    | evaluate ipv4_lookup(GeoIPDB_FULL, IpAddress, network);
WindowsEvents

The result was a data set where every failed logon against the honeypot now included both the attacker IP address and the country it originated from, a critical step for threat hunting and reporting.

Part Six: Sentinel Workbook Attack Map

With the enriched data in place, I created an attack map that displays global brute force activity against the honeypot VM.

  • Opened the Workbooks area in Sentinel and created a new blank Workbook.
  • Removed prepopulated visual elements to start with a clean canvas.
  • Added a Query element and switched to the advanced editor.
  • Pasted a JSON configuration that defines the query and the map visualization settings.
  • Bound the latitude and longitude fields to columns produced by the GeoIP enrichment query.
  • Adjusted map style, bubble intensity, and labels for readability.

Once saved, the Workbook updates as new SecurityEvent records are ingested. The result is a live map where each bubble represents a cluster of failed logon attempts from a particular region, giving a clear visual of where malicious activity is concentrated.

Key Insights
  • The honeypot began receiving inbound attacks shortly after exposure, confirming that misconfigured RDP hosts are rapidly targeted on the internet.
  • Most attacks originated from a small set of countries and hosting providers, which is consistent with known brute force infrastructure.
  • Common username attempts included admin, test, guest, and generic employee style names.
  • Using KQL with GeoIP enrichment made it easy to pivot from a single IP address to a broader view of attacker infrastructure.
  • Sentinel Workbooks provided a simple way to present complex data through an intuitive attack map for stakeholders.
