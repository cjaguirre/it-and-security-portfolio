What I built

A Windows VM was deployed in Azure, given a public IP, and intentionally weakened so internet scanners would find and attack it. Every failed RDP login flowed into Log Analytics and Microsoft Sentinel, where KQL and GeoIP enrichment turned raw event logs into a map-ready view of global attack activity.

Key outcomes

  • • Windows honeypot exposed and attracting attacks within minutes
  • • Full log pipeline: VM → Log Analytics → Microsoft Sentinel
  • • GeoIP watchlist enabling IP-to-country enrichment via KQL
  • • Interactive Sentinel workbook map of all failed logon events
  • • 72 correlated incidents grouped from 4,623 raw events

Lab Walkthrough

Phases 1–2 — Azure Setup & Honeypot Exposure

The subscription was validated first to make sure the full pipeline — VM, Log Analytics, Sentinel — could run without quota issues. Then a Windows VM was deployed and deliberately weakened: firewall off, NSG wide open, public RDP on. The goal was to make it look like an easy target so real attacks would come quickly.

Making it a target

  • • NSG configured to allow inbound traffic from any source
  • • Windows firewall disabled across all profiles
  • • Weak decoy account created to generate early failed logons
  • • First external connection attempts appeared within minutes

Phase 3 — Log Ingestion & Sentinel Setup

Once attacks started rolling in, the next step was capturing them. A Log Analytics workspace was created, Sentinel was enabled on top of it, and the Azure Monitor Agent was installed on the honeypot to stream SecurityEvent logs continuously. Event ID 4625 — failed logon — became the primary signal.

The log pipeline

  • • Log Analytics workspace created and linked to the honeypot resource group
  • • Sentinel enabled on the workspace for SIEM-style analysis
  • • AMA installed on the VM via the Windows Security Events connector
  • • SecurityEvent table confirmed populating with 4625 events

Phase 4 — GeoIP Enrichment with KQL

Raw IP addresses don't tell a story on their own. A GeoIP CSV was imported as a Sentinel watchlist, then KQL's ipv4_lookup() joined each failed logon IP to its country, region, and coordinates — turning a flat event table into map-ready threat intelligence.

How the enrichment works

  1. GeoIP CSV (IPv4 ranges + country/lat/long) uploaded as a Sentinel watchlist
  2. KQL filters SecurityEvent to Event ID 4625 only
  3. ipv4_lookup() joins each IP to the watchlist
  4. Output: enriched table with country, region, coordinates per event

Phase 4 — Attack Map Workbook

The enriched KQL output was wired into a Sentinel workbook with an Azure Maps canvas. Each point on the map represents failed RDP logons from a specific location — size and color reflect event volume. The result is a view that works for both a SOC analyst and a non-technical executive.

What the map shows

  • • Every enriched 4625 event plotted by geographic source
  • • Dot size reflects attack volume from that location
  • • Filterable by time range, username, and source country
  • • Highlights regional hotspots generating the most persistent traffic

Phase 5 — SOC Investigation

With the map and enriched data in place, the final phase was working through it like a SOC analyst — pivoting between IPs, grouping events into incidents, and figuring out what the attack patterns actually tell you about attacker behavior and detection opportunities.

What the data revealed

  • • Attacks started within minutes of the VM going live — RDP exposure is discovered fast
  • • Most attempts came from automated infrastructure, not individual humans
  • • A small number of IPs accounted for a large share of total attempts
  • • Persistent attackers retried over multiple days — not one-off scans
  • • 72 distinct campaigns could be identified from 4,623 raw events

RAPID DISCOVERY

AUTOMATED CAMPAIGNS

GLOBAL DISTRIBUTION

GEOIP VALUE

DETECTION TUNING

PLAYBOOK POTENTIAL

The KQL queries and workbook provide a direct foundation for automated response playbooks that block abusive IPs.

Tier 1 — Reduce Exposure

Eliminate the configurations that make brute-force viable.

  • Retire public RDPReplace with Just-in-Time access, VPN, or Azure Bastion.
  • Enforce NSG restrictionsLock inbound rules to trusted IP ranges — not any source.
  • Require MFA everywhereCredential stuffing fails if a second factor is always required.

Tier 2 — Detection & Analytics

Turn noisy telemetry into actionable detection logic.

  • Sentinel analytic rulesTrigger on failed-logon patterns and GeoIP anomalies from unusual sources.
  • Promote KQL queriesMove proven enrichment queries into reusable hunting workbooks.
  • Add threat intel feedsLayer known-bad infrastructure data on top of GeoIP for faster triage.

Tier 3 — Strategic Resilience

Build repeatable practices that scale.

  • Automated playbooksAuto-block abusive IPs or notify the SOC when campaigns cross thresholds.
  • Scheduled exposure scanningContinuously check for re-exposed RDP or misconfigured NSG rules.
  • Reduce legacy attack surfacePhase out RDP and other legacy remote access patterns over time.

What I Learned

Key takeaways

  • • Exposed RDP gets found and attacked almost immediately — there's no quiet period
  • • Most brute-force is automated, which makes it highly detectable with the right KQL rules
  • • GeoIP enrichment turns a wall of IP addresses into something anyone can understand
  • • Workbooks and KQL together make it easy to tell the story of what's attacking you and from where
  • • 4,623 events collapsed into 72 incidents — good correlation logic matters for triage speed
