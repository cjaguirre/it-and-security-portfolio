Azure Sentinel Threat Analysis

Designed and deployed a Windows honeypot in Azure to capture and analyze global attack traffic. Forwarded event logs into Microsoft Sentinel, enriched malicious IPs with threat intelligence, and visualized real time attacker activity through an interactive world map.

Realistic SOC workflow simulation Tools: Azure, Log Analytics, Sentinel, KQL

Project Narrative

I created a simple, real-world security investigation by setting up a Windows computer in the cloud and leaving its remote access port open on purpose. This acted as a “honeypot,” attracting automated attackers from around the world.

As the attacks came in, the system recorded every failed login attempt. I then added location information to each attacker’s IP address and turned all the activity into clear visual charts using Microsoft Sentinel.

This project shows what happens when a server is left unprotected on the internet. Scanners quickly find it, attempt to break in, and generate a steady stream of security alerts. The lab demonstrates how an analyst collects, enriches, and visualizes this data to understand what’s happening and respond effectively.

What I Built

  • Set up an Azure account and all required resources.
  • Deployed a Windows 10 honeypot that was fully open to the internet.
  • Configured the network rules to allow all traffic in.
  • Turned off the Windows firewall to make the system intentionally vulnerable.
  • Connected the VM to a Log Analytics Workspace to collect live security logs.
  • Added the VM into Microsoft Sentinel for monitoring and analysis.
  • Uploaded a GeoIP file to identify where attacks were coming from.
  • Used a KQL function to match attacker IPs with their location.
  • Built a global attack map showing over 4,600 enriched security events.

Lab breakdown

1. Azure subscription setup

Verified access to VMs, NSGs, Log Analytics, and Microsoft Sentinel.

I started by confirming that my Azure tenant could deploy virtual machines, create Network Security Groups, build a Log Analytics Workspace, and onboard Microsoft Sentinel. This matches what a junior SOC analyst would inherit in a real environment.

Azure home overview with core services surfaced.

Subscription view confirming resource availability.

2. Honeypot VM and intentional exposure

Created a Windows 10 VM, opened RDP, disabled firewall, and relaxed NSG rules.

The objective here was not to harden the host, but to safely recreate a misconfiguration. I deployed a Windows 10 VM with a public IP, opened its inbound NSG rules, and disabled the Windows firewall to invite real world brute force traffic.

  • Provisioned a Windows 10 VM from the gallery with RDP access enabled.
  • NSG updated to allow inbound traffic from any source.
  • All Windows firewall profiles disabled through wf.msc.
  • Generated baseline failed logons using a decoy account (for example employee).

NSG rule intentionally configured to allow all inbound traffic.

3. Log forwarding and Microsoft Sentinel integration

Logs ingested via Azure Monitor Agent into Log Analytics and surfaced in Sentinel.

Once the honeypot was generating SecurityEvent logs locally, I wired it into Azure monitoring so Sentinel could query and visualize the data.

  • Created a Log Analytics Workspace as the central store for Windows logs.
  • Enabled Microsoft Sentinel on that workspace.
  • Configured the Windows Security Events via AMA connector to install the Azure Monitor Agent and create a data collection rule.
  • Verified ingestion by querying SecurityEvent for Event ID 4625 (failed logons).

AMA connector showing a healthy connection from VM to Log Analytics.

4. GeoIP watchlist and KQL enrichment

GeoIP dataset imported as a watchlist and joined to SecurityEvent logs using KQL.

To make attacker locations obvious, I imported a summarized GeoIP dataset and used Kusto Query Language to enrich failed logons with geographic attributes.

  • Loaded a CSV mapping IPv4 ranges to countries and regions.
  • Created a Sentinel watchlist named geoip using network as the key.
  • Used ipv4_lookup to join 4625 events with GeoIP data.

GeoIP watchlist loaded into Sentinel with country and region fields.

5. Sentinel workbook attack map

Workbook built to visualize attack origin points from enriched 4625 events.

Finally, I created a Sentinel workbook that runs the enriched KQL query and plots attacker IPs on an Azure map visualization. Filters and time controls let me analyze different attack windows.

  • New workbook created from a blank canvas.
  • Query visualization configured to run the GeoIP enriched KQL.
  • Latitude and longitude fields mapped to the map visual.
  • Controls added for time range and dynamic filtering.

Sentinel workbook map showing global failed logon activity.

Sample KQL

Baseline failed logons 

SecurityEvent
| where EventID == 4625
| sort by TimeGenerated desc

GeoIP enriched 4625 events 

let GeoIPDB_FULL = _GetWatchlist("geoip");
let WindowsEvents =
    SecurityEvent
    | where EventID == 4625
    | sort by TimeGenerated desc
    | evaluate ipv4_lookup(GeoIPDB_FULL, IpAddress, network);
WindowsEvents

Architecture overview

Components

  • Public internet — real world brute force traffic sources.
  • Azure NSG (allow all inbound) — intentionally exposed perimeter.
  • Windows honeypot VM — generates SecurityEvent telemetry.
  • Azure Monitor Agent — forwards logs to Log Analytics.
  • Log Analytics Workspace — central store plus GeoIP reference data.
  • Microsoft Sentinel — SIEM used for KQL analysis and visualization.

Data flow

  • Internet → VM — attackers launch automated RDP scans.
  • VM → Log Analytics — AMA ships Windows SecurityEvent logs.
  • Log Analytics → Sentinel — SIEM detection and analytics.
  • GeoIP watchlist → KQL — IP ranges joined to enrich attacker events.
  • Sentinel workbook → analyst — enriched events visualized on a global map.

Key insights

  • Exposed RDP hosts attract brute force traffic within minutes of going online.
  • Most attacker infrastructure is automated and hosted, not individual users.
  • GeoIP enrichment immediately reveals global attack clusters and patterns.
  • Sentinel workbooks convert raw logs into visuals that are easy to brief and share.

Summary

This project ties together endpoint logging, Azure monitoring, and SIEM analysis into one pipeline. I stood up a deliberately vulnerable Windows host, collected its logs in Log Analytics, analyzed the resulting events in Microsoft Sentinel using KQL, and layered on GeoIP enrichment to understand where attacks were coming from. The final attack map presents hostile activity in a format that analysts, managers, and non technical stakeholders can all understand while still reflecting the underlying technical work required to operate a modern SOC.
