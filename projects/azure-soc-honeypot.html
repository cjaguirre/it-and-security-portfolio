Azure SOC Honeypot and Attack Map Lab

Built a Windows honeypot in Azure, intentionally exposed it to the internet, collected real-world brute force activity, forwarded logs into Microsoft Sentinel, enriched attacker IPs with GeoIP data, and visualized global attack patterns on an interactive map.

Built as a realistic SOC-style workflow Tools: Azure, Log Analytics, Sentinel, KQL

Project Narrative

End to end SOC style workflow

The goal of this lab was to recreate what happens in a Security Operations Center when a misconfigured server is left exposed. I deployed a Windows 10 honeypot VM in Azure, opened its inbound security controls, and allowed the public internet to attack it.

Windows SecurityEvent logs were sent to a Log Analytics Workspace via the Azure Monitor Agent and then surfaced in Microsoft Sentinel. From there I used Kusto Query Language to focus on failed logons, joined those events with a GeoIP watchlist, and turned the results into an attack map that tracks where brute force attempts originate.

What I Built

High level checklist

  • Configured an Azure subscription and portal environment.
  • Deployed a Windows 10 honeypot VM with a public IP.
  • Opened inbound access in the Network Security Group to invite real attacks.
  • Disabled Windows Firewall profiles on the VM to expose RDP.
  • Forwarded Windows SecurityEvent logs to a Log Analytics Workspace via Azure Monitor Agent.
  • Onboarded the workspace into Microsoft Sentinel as the SIEM layer.
  • Imported a GeoIP CSV file into Sentinel as a watchlist for location data.
  • Used KQL and ipv4_lookup to enrich failed logons with GeoIP context.
  • Created a Sentinel Workbook that visualizes attacker locations on a world map.

Lab Breakdown

1. Azure Subscription Setup

I created an Azure subscription, signed into the portal, and verified that the tenant could deploy virtual machines, Network Security Groups, a Log Analytics Workspace, and a Microsoft Sentinel instance. This mirrors the initial access a junior SOC analyst would receive in a real environment.

Azure Home Overview

Azure Subscription Overview

2. Honeypot VM and Intentional Exposure

Next, I deployed a Windows 10 VM with a public IP. The goal here was not to harden the host, but to expose it in a controlled lab so the internet would naturally attack it.

  • Created a Windows 10 VM from the gallery with RDP access enabled.
  • Modified the VM’s Network Security Group to allow all inbound traffic from any source.
  • Logged into the VM and disabled all Windows Firewall profiles through wf.msc.
  • Performed a few failed logon attempts using a decoy username (for example, employee) to generate baseline events.

NSG Rule: Allow All Inbound

3. Log Forwarding and Microsoft Sentinel Integration

After confirming the VM was producing SecurityEvent logs locally, I wired it into Azure’s monitoring pipeline.

  • Created a Log Analytics Workspace to act as the central log store.
  • Enabled Microsoft Sentinel on that workspace.
  • Configured the “Windows Security Events via AMA” connector, which installed the Azure Monitor Agent extension and created a Data Collection Rule.
  • Validated ingestion by querying the SecurityEvent table for failed logons (Event ID 4625).

AMA Connector: Windows Security Events (Connected)

4. GeoIP Watchlist and KQL Enrichment

SecurityEvent logs include attacker IP addresses but not geography. To add that context, I imported a GeoIP dataset as a Sentinel watchlist and used KQL to join the two.

  • Downloaded a summarized GeoIP CSV that maps IPv4 ranges to countries and regions.
  • Created a Sentinel watchlist named geoip using network as the search key.
  • Used KQL with ipv4_lookup to enrich 4625 events with GeoIP attributes.

Sentinel Watchlist: GeoIP Rows

5. Sentinel Workbook Attack Map

Finally, I built a Workbook that runs a KQL query against the enriched events and plots attacker locations on an Azure map.

  • Created a new Workbook in Sentinel and started from a blank canvas.
  • Added a Query visualization, then used the advanced editor to define both the query and map settings.
  • Bound latitude and longitude fields from the GeoIP data so each failed logon appears as a point on the globe.
  • Configured filters and time range controls to explore attack patterns over different windows.

Sentinel Workbook: Global Attack Map Visualization

Sample KQL

Baseline failed logons 

SecurityEvent
| where EventID == 4625
| sort by TimeGenerated desc

GeoIP-enriched 4625 events 

let GeoIPDB_FULL = _GetWatchlist("geoip");
let WindowsEvents =
    SecurityEvent
    | where EventID == 4625
    | sort by TimeGenerated desc
    | evaluate ipv4_lookup(GeoIPDB_FULL, IpAddress, network);

Architecture Overview

Components

  • Public Internet — real-world brute-force traffic sources.
  • Azure NSG (Allow All Inbound) — intentionally exposed perimeter.
  • Windows Honeypot VM — generates SecurityEvent telemetry.
  • Azure Monitor Agent — forwards logs to Log Analytics.
  • Log Analytics Workspace — central store + GeoIP reference data.
  • Microsoft Sentinel — SIEM used for KQL analysis & visualization.

Data Flow

  • Internet → VM — attackers launch automated RDP scans.
  • VM → Log Analytics — AMA ships Windows SecurityEvent logs.
  • Log Analytics → Sentinel — SIEM detection & analytics processing.
  • GeoIP Watchlist → KQL — IP ranges joined to enrich attacker events.
  • Sentinel Workbook → Analyst — enriched events visualized on global map.

Key Insights

  • Exposed RDP hosts attract brute-force traffic within minutes of being online.
  • Most attacks originate from known automated infrastructure, not individuals.
  • GeoIP enrichment instantly reveals global attack clusters and patterns.
  • Sentinel Workbooks turn raw telemetry into clear, analyst-ready visuals.

Summary

This lab ties together endpoint logging, Azure monitoring, and SIEM analysis into one workflow. I stood up a vulnerable Windows host, collected its logs in Log Analytics, analyzed those events in Microsoft Sentinel using KQL, and layered on GeoIP enrichment to understand where attacks were coming from. The finished attack map shows hostile activity in a way that analysts, managers, and non-technical stakeholders can all understand, while still highlighting the underlying technical work that goes into building a modern SOC pipeline.
