Azure Sentinel Threat Analysis

Designed and deployed a Windows honeypot in Azure to capture global attack traffic. Forwarded logs into Microsoft Sentinel, enriched attacker IPs with threat intel, and built an interactive world map to visualize real time security events.

SOC Investigation Workflow Tech Stack: Azure VM, Sentinel, Log Analytics, KQL, Watchlists

Project Narrative

I deployed a Windows VM in Azure, opened it fully to the internet, and turned off protections so it would attract automated attackers. As attacks hit the system, each failed login attempt and source IP was logged into Microsoft Sentinel.

I then enriched attacker IPs with geolocation data and visualized everything in charts and a global threat map. This project shows how quickly an exposed machine is found and exploited online, and how SOC analysts investigate and interpret that data.

What I Built

  • Deployed an Azure Windows honeypot fully exposed to the internet
  • Configured networking + disabled protections to capture real attack traffic
  • Connected the VM to Log Analytics and Microsoft Sentinel
  • Enriched attacker IPs with GeoIP data using a custom KQL function
  • Built an interactive global map visualizing 4,600+ security events

2. Honeypot VM and Intentional Exposure

Deployed a Windows 10 VM, opened RDP, and removed default protections

To capture real attacker activity, I deployed a Windows 10 VM with a public IP and intentionally relaxed its security. Opening RDP, loosening the NSG rules, and disabling the Windows firewall made the VM easy to find and target—creating a steady stream of authentic brute-force attempts for analysis in Sentinel.

Key Steps:

  • Deployed a Windows 10 VM with RDP enabled.
  • Allowed inbound traffic from any source through the NSG
  • Disabled all Windows firewall profiles
  • Generated initial failed logons using a decoy account.

These changes ensured the VM was highly visible to internet scanners and automated attack traffic.

3. Log Forwarding and Microsoft Sentinel Integration

Forwarded Windows security logs to Log Analytics and connected them to Sentinel

Once the honeypot began generating failed logons, I connected the VM to Azure’s monitoring pipeline so all activity could be captured and analyzed in one place. Log Analytics handled the ingestion, and Microsoft Sentinel provided the analysis and visualization layer.

Key Steps:

  • Set up a Log Analytics Workspace as the central log collector.
  • Enabled Microsoft Sentinel on that workspace.
  • Installed the Azure Monitor Agent using the Windows Security Events via AMA connector.
  • Verified that logs were flowing by querying Event ID 4625 (failed logons).

After the AMA pipeline was active, Log Analytics immediately began receiving authentication failures from the exposed VM, giving Sentinel a real time feed of attacker activity.

4. GeoIP Watchlist and KQL Enrichment

Added location data to failed-logon events using a GeoIP watchlist

Once logs were flowing into Sentinel, I enriched the attacker IPs with geographic data to make the activity easier to interpret. A GeoIP CSV was imported as a Sentinel watchlist, and KQL was used to match each failed logon with its country and region.

Key Steps:

  • Uploaded a CSV containing IPv4 ranges with country and region fields.
  • Created a Sentinel watchlist named geoip.
  • Used ipv4_lookup in KQL to join the watchlist with Event ID 4625 logs.

This turned raw IP addresses into clear, location-based insights for the attack map.

5. Sentinel Workbook Attack Map

Visualized attacker locations using GeoIP-enriched logon events

With the GeoIP enrichment complete, I used a Sentinel workbook to turn the failed-logon data into a world map of attacker activity. By connecting the workbook to the enriched KQL query, each event could be placed on the map using latitude and longitude, making it easy to see where attacks were coming from.

Key Steps:

  • Started a blank Sentinel workbook and added a KQL visual.
  • Connected it to the GeoIP-enhanced SecurityEvent query.
  • Mapped latitude and longitude fields to Azure Maps.
  • Added filters for time ranges and event counts.

The final workbook shows real-time attacker distribution, highlighting which regions generated the most persistent brute-force attempts.

Sentinel workbook map showing global failed-logon activity from enriched SecurityEvent logs.

Sample KQL

These queries were used to analyze failed RDP logons during the lab. The first retrieves raw failed-logon events, and the second enriches them with GeoIP data so attacker locations can be mapped and visualized.

Baseline failed logons 

SecurityEvent
| where EventID == 4625
| sort by TimeGenerated desc

GeoIP enriched 4625 events 

let GeoIPDB_FULL = _GetWatchlist("geoip");
let WindowsEvents =
    SecurityEvent
    | where EventID == 4625
    | sort by TimeGenerated desc
    | evaluate ipv4_lookup(GeoIPDB_FULL, IpAddress, network);

Section Overview

Components

  • Public Internet — Real attackers scanning and brute-forcing exposed services.
  • Azure NSG (Open Inbound) — Deliberately left wide-open to attract traffic.
  • Windows Honeypot VM — Generates SecurityEvent logs from failed RDP attempts.
  • Azure Monitor Agent — Forwards SecurityEvent logs to Log Analytics.
  • Log Analytics Workspace — Central storage and query engine for collected logs.
  • Microsoft Sentinel — Runs KQL, enriches data, and visualizes attacker behavior.

Data flow

  • Scanners → Honeypot VM → Failed RDP events generated.
  • Azure Monitor Agent → Log Analytics ingestion.
  • Log Analytics → Sentinel for parsing and investigation.
  • GeoIP watchlist → KQL lookup → Country and region enrichment.
  • Sentinel workbook → Global attack-map visualization.

Key insights

  • Exposed RDP services attract scanning and brute-force activity within minutes of going online.
  • Most attempts originate from automated infrastructure rather than human attackers.
  • GeoIP enrichment immediately reveals geographic patterns and regional concentrations of activity.
  • Mapping enriched events provides faster insight than reviewing raw logs, highlighting trends at a glance.

Summary

This project demonstrates how real attack telemetry can be captured, enriched, and visualized using Azure’s security stack. By deploying an intentionally exposed Windows host, collecting its logs, and combining them with GeoIP enrichment in Sentinel, I created a streamlined investigation workflow that surfaces global brute-force activity in a clear and accessible format. The final workbook makes it easy for both technical and non-technical audiences to understand where attacks originate and how they unfold in a cloud environment.
