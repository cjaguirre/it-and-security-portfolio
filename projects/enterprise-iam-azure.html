Enterprise Identity & Access Management Lab

End-to-end identity lifecycle management in Microsoft Entra ID — provisioning users across departments, enforcing RBAC, managing external B2B collaboration, executing help desk workflows, and validating security posture through audit logs and authentication monitoring.

Project overview

Deployed a full enterprise IAM environment inside Microsoft Entra ID to simulate real-world identity operations from the ground up — tenant setup, user provisioning, group-based access control, external collaboration, help desk operations, MFA enforcement, and audit log analysis, all translated into a structured hardening strategy.

Key deliverables

  • 6 internal users provisioned across IT, HR, and Finance departments
  • 3 security groups implementing RBAC access governance
  • B2B guest identity onboarded with attribute enrichment and scoped access
  • Help desk operations — password reset, account suspension, and user restore
  • MFA enforcement and authentication method review across users
  • Sign-in and audit log analysis for event investigation and compliance
  • IAM architecture diagram documenting the complete identity flow

Lab Breakdown

Phase 1–2 — Environment Setup and Identity Awareness

Before any provisioning, the Entra Admin Center was accessed and explored to understand the full IAM surface. The tenant identity model was reviewed to distinguish cloud, on-premises AD, and hybrid identity types — foundational knowledge for the lab phases ahead.

1.1 — Accessing the Entra Admin Center

Signed into the Microsoft Entra Admin Center to establish access to the enterprise identity control plane. The dashboard centralizes all identity management capabilities — users, groups, roles, monitoring, and security — from a single portal.

  • Authenticated to entra.microsoft.com with admin credentials
  • Confirmed tenant name and dashboard accessibility

Entra ID overview dashboard — admin access confirmed to the tenant identity control plane.

1.2 — Exploring IAM Navigation Surfaces

Navigated through Identity, Monitoring, and Security sections to familiarize with available IAM surfaces before making any changes. This scoping step ensures deliberate, informed action across all subsequent phases.

  • Surfaces explored: Users, Groups, Roles & Admins, Sign-in Logs, Audit Logs, Conditional Access
  • No changes — awareness and orientation only

Active Directory user selection menu confirming navigation across identity management surfaces.

Phase 3 — Internal User Provisioning

Six internal accounts were created across IT, HR, and Finance to simulate enterprise employee onboarding. Each account was configured with an auto-generated password, forced reset on first sign-in, and a usage location set for licensing compliance.

3.1 — IT Department (it.admin, it.support)

The IT Admin and IT Support accounts were provisioned first to establish the IT department's identity footprint. Admin accounts carry elevated access requirements and will receive stricter authentication controls.

  • Identity → Users → New user → Create new user
  • Auto password with force-reset applied to both accounts
  • Usage location configured for licensing compliance

it.admin account created with password policy and location configured.

it.support provisioned with matching enforced password reset policy.

3.2 — HR Department (hr.manager)

The HR Manager account was provisioned to represent the HR department's management tier. HR identities will be assigned to the HR-Team security group and scoped to HR-relevant resources via RBAC.

hr.manager provisioned — HR department management-tier identity.

3.3 — Finance Department (fin.analyst)

The Finance Analyst account was provisioned to represent a high-sensitivity departmental identity. Finance accounts carry elevated data governance requirements given the sensitivity of the resources they access.

fin.analyst created — high-sensitivity Finance department identity.

3.4 — Validating the Complete User Roster

After all six accounts were created, the directory user list was reviewed to confirm every identity was correctly reflected — accurate UPNs, display names, and department metadata across all provisioned accounts.

  • Confirmed: it.admin, it.support, hr.manager, hr.staff, fin.analyst, fin.staff
  • UPNs, directory presence, and attribute population verified for each

Complete user roster — all six departmental identities confirmed in the directory.

3.5 — Reviewing Identity Details and Properties

Individual user profiles were opened to verify identity metadata — job title, department, and usage location — ensuring attributes were fully populated for accurate RBAC targeting downstream.

Identity details panel showing complete attribute profile for a provisioned internal account.

Phase 4 — External Collaboration (B2B Guest Identity)

Phase 4 simulates enterprise B2B collaboration by onboarding an external contractor through Entra's guest invitation workflow. Guest identities maintain a clear separation from internal member accounts via the #EXT# designation in their UPN and a Guest user type in the directory.

4.1 — Inviting the External Contractor

The guest invitation was initiated via Users → New user → Invite external user. An invitation email was dispatched to the external address. Once accepted, the account appears in the directory with a #EXT# suffix and Guest user type — visually and functionally distinct from internal identities.

  • Invitation sent to personal email — simulating a contractor onboarding scenario
  • Guest account confirmed with #EXT# suffix in UPN
  • User type set to Guest — clearly separated from Member identities

External user invitation form — initiating the B2B guest onboarding workflow.

4.2 — Reviewing the Invitation Message

The invitation message was reviewed before dispatch to confirm the communication that external collaborators receive — including tenant context and access instructions.

Invitation message preview confirming B2B onboarding communication sent to external user.

4.3 — Enriching the Guest Profile and Confirming Access Scope

After the invitation, the guest account was enriched with organizational metadata and reviewed to verify EXT designation and Guest user type — confirming clean separation from internal member identities.

  • Job title, department, and usage location added to guest properties
  • EXT designation and Guest user type confirmed in directory

Guest account review — EXT designation and Guest user type confirmed.

Contractor identity profile showing enriched metadata and scoped guest access.

Phase 5 — Access Governance (RBAC Groups and Role Assignments)

Three departmental security groups formed the RBAC foundation. Users were mapped to their respective groups, and role assignments were reviewed to validate least-privilege access boundaries across the directory. Group-based access control scales governance without per-user complexity.

5.1 — Creating Security Groups

IT-Team, HR-Team, and Finance-Team security groups were created — each representing a departmental access boundary. Group type set to Security with manual membership assignment.

  • Identity → Groups → New group → Security type
  • Groups created: IT-Team, HR-Team, Finance-Team

All three departmental security groups — the RBAC structure foundation.

5.2 — Assigning Users to Groups

Internal users were mapped to their departmental groups. Group membership is the mechanism through which access permissions propagate — users inherit the access rights of every group they belong to, keeping permissions manageable at scale.

Users assigned to departmental groups — access inheritance via group membership confirmed.

5.3 — Reviewing Role Assignments and Permissions

The directory role assignment list and assigned permissions overview were reviewed to confirm access was correctly bounded by least privilege — no account held permissions beyond what their role required.

Assigned roles and permissions — least-privilege access scope confirmed.

User access permissions grid validating role-to-user mapping across the directory.

Phase 6 — Help Desk Operations

Common Tier 1 identity lifecycle workflows were executed — password resets, account suspension, and user restore — demonstrating the most frequent day-to-day operations in any enterprise IAM environment.

6.1 — Password Reset

A password reset was performed to simulate a Tier 1 support ticket. The operation generates a temporary credential with a forced change on next sign-in, minimizing the exposure window of the temp password.

  • Opened target user → Reset password
  • Temporary password generated with forced reset enforced
  • End-user password update prompt confirmed the complete cycle

Password reset portal — Tier 1 credential management workflow from admin side.

End-user password update prompt — completing the admin-to-user reset cycle.

6.2 — User Properties Review Before Lifecycle Actions

Full attribute and security profile reviews were performed before any account status changes — confirming current state, MFA enrollment, and authentication methods as part of a responsible lifecycle management workflow.

Full attribute list confirming identity metadata state before lifecycle modification.

Phase 7–8 — Authentication Monitoring and Audit Investigation

MFA was enforced on privileged accounts, sign-in telemetry was generated and reviewed, and audit logs were analyzed to confirm full event traceability — a critical capability for compliance reviews, security investigations, and ongoing threat detection.

7.1 — Admin Account Security Info Review

The admin account's security information was reviewed to verify MFA registration status and authentication method coverage — the baseline check before generating sign-in activity.

Admin account security info — MFA registration and method coverage confirmed.

7.2 — MFA Enforcement on Admin Accounts

MFA was configured and enforced on admin-tier accounts — simulating the authentication hardening applied to privileged identities in enterprise environments. A compromised admin credential without MFA represents one of the highest-impact breach scenarios in cloud IAM.

  • MFA enforcement applied to admin-tier accounts
  • Authentication method registered and verified

MFA enforcement configured on admin account — authentication hardening applied to privileged identities.

7.3 — Authentication Method Status Across Users

Authentication method status was reviewed across all user accounts to identify any identities lacking MFA enrollment — accounts without secondary authentication represent the most exposed gap in any enterprise identity environment.

Authentication methods status — MFA enrollment coverage reviewed across all user accounts.

The system activity log was reviewed to capture all administrative events generated across the lab — user creation, group assignments, role changes, and help desk actions — providing a complete, timestamped record for compliance review.

  • Events captured: user creates, password resets, group membership changes
  • Metadata reviewed: timestamps, initiating actor, target object

System activity log capturing all administrative events across the lab workflow.

8.2 — Audit Log User Activity Report

The audit log was used to investigate specific administrative events — confirming the chain of actions, their initiating actor, and timestamps. This directly mirrors the investigation workflow used during real security incidents or compliance audits.

Audit log user activity report — event traceability confirmed for compliance and investigation.

IAM Hardening Strategy

Tier 1 — Identity Control

Enforce strict identity governance and close overprivileged access paths.

  • •MFA EverywhereRequire MFA for all users, not just admin accounts.
  • •Least Privilege RBACAudit and remove excessive role assignments regularly.
  • •Privileged Identity ManagementEnable PIM for just-in-time admin elevation.

Tier 2 — Access Hardening

Restrict the attack surface and enforce policy-driven access controls.

  • •Conditional Access PoliciesBlock legacy auth protocols, require compliant devices.
  • •Guest Access ScopingRestrict external identities to explicitly approved resources.
  • •Self-Service Password ResetEnable SSPR to reduce credential reset volume at help desk.

Tier 3 — Continuous Visibility

Maintain ongoing identity visibility to detect risk and prevent regression.

  • •Risky Sign-in AlertsAlert on anomalous sign-ins and impossible travel events.
  • •Audit Log RetentionConfigure retention and export to SIEM for investigation.
  • •Secure Score ReviewsTrack identity secure score on a recurring cycle.

Key Insights

  • Group-based RBAC scales access governance without per-user management complexity.
  • Audit logging enables full event traceability — essential for compliance and incident investigation.
  • Enforcing MFA on privileged accounts significantly narrows the credential compromise attack surface.
  • Separating guest identities with EXT designation maintains clean access boundaries during external collaboration.
