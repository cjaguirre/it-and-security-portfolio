Phase 5 — RBAC Security Groups

Three department-based security groups give every user a defined access boundary. Assigning permissions at the group level — not per user — keeps the model auditable and easy to maintain as the org changes. The external guest sits in IT-Team alongside internal IT accounts, demonstrating that B2B users can be governed within the same RBAC model.

Three groups, three boundaries

  • • IT-Team — it.admin, it.support, Contractor (Guest)
  • • HR-Team — hr.manager, hr.staff
  • • Finance-Team — fin.analyst, fin.staff
  • • All groups use Assigned membership — no dynamic rules

Security Posture & Hardening

The Identity Secure Score came in at 54.17% with 5 open recommendations. The biggest gap: no Conditional Access policies are active. That's the highest-priority fix — without it, anyone with valid credentials can sign in from anywhere with no additional verification. The hardening plan below addresses the score gaps in order of impact.

Identity Control

Strengthen the identity layer itself.

  • Extend MFA to all usersSecure Score flags this open — admin MFA isn't enough, member identities need it too.
  • Add a second global adminOnly one exists — creates a lockout risk if that account is compromised.
  • Tighten it.admin rolesUser Administrator is broad — review whether a scoped role is more appropriate.

Access Hardening

Control how and where access is granted.

  • Create Conditional Access policiesPriority one: require MFA outside the corporate network, block legacy auth protocols.
  • Scope guest accessContractor Test is in IT-Team with no resource restrictions — limit to explicitly approved apps.
  • Restrict user app consentSecure Score recommends blocking users from consenting to unverified third-party apps.

Continuous Visibility

Keep monitoring sharp over time.

  • Export logs to a SIEMPortal review isn't enough — configure diagnostic settings to retain logs externally.
  • Alert on risky sign-insSign-in failures happened with no alert triggered — enable Identity Protection alerting.
  • Track score over timeScore history goes back to Nov 2025 — set a recurring review to measure improvement.

What I Learned

Key takeaways

  • • Every provisioning decision creates an audit trail — roles and groups should be intentional from day one
  • • B2B guests can be governed the same as internal users, but their scope needs to be explicitly defined
  • • A Secure Score in the 50s usually means Conditional Access isn't set up — that's the first thing to fix
  • • Audit logs are only useful if someone is actually looking at them — log export and alerting matter
  • • Help desk operations like suspend and restore work cleanly, but they need to be tested, not assumed
