Windows Sysmon Threat Detection Lab

Built and tuned a single host Windows monitoring lab around Sysmon to practice endpoint telemetry, threat hunting, and SOC style reporting. Configured a realistic workstation, deployed Sysmon with a community configuration, focused on high value event IDs, simulated attacker behavior, and documented investigations with MITRE ATT&CK mapping.

Endpoint telemetry · Windows logging Tools: Sysmon, Event Viewer, PowerShell, MITRE ATT&CK Use case: Single host detection lab

Project Narrative

I approached this lab as if I were a junior SOC analyst given one Windows endpoint to monitor. The goal was to treat Sysmon telemetry as a story: prepare a clean baseline, add tuned logging, then follow how the host changes as simulated attacker behavior is introduced.

Starting from a fresh Windows 10 VM, I locked down privacy, tuned the hardware profile, and built a comfortable analysis workspace with PowerShell and Event Viewer. I then deployed Sysmon using a community configuration similar to what blue teams use in production, verified that events were flowing, and narrowed in on high signal event IDs for process, file, registry, DNS, and network activity.

From there I walked through investigations that mirror real cases: LOLBin downloads, Run key persistence, encoded PowerShell commands, and command and control style traffic. Each scenario ends in a SOC ready write up that explains what happened, how it was detected, and which MITRE ATT&CK techniques apply.

What I Built

  • A Windows 10 monitoring VM tuned for log analysis and low background noise.

With telemetry tuned, I introduced realistic behavior in stages: normal browsing, file creation, LOLBin downloads, registry persistence, and command and control style traffic. Each screenshot represents a step in a story that can be retold using Sysmon logs alone.

5. SOC Reporting and MITRE ATT&CK Mapping

Turning Sysmon telemetry into structured incidents and technique coverage

The final phase focuses on reporting. I treated each simulation as a ticket: summarize what happened, list the key commands and events, explain why it is suspicious, and tie it to MITRE ATT&CK. The goal is for someone who did not run the lab to still understand the full incident story.

Post exploitation whoami usage, revisited as part of a broader discovery and persistence sequence.

systeminfo execution recorded as another process creation event, filling out the discovery phase.

I then wrapped the persistence scenario into a SOC style template with fields for indicators, event IDs, registry paths, payload details, and mapped techniques. This is the type of artifact that can be linked directly into a ticketing system or used for onboarding new analysts.

SOC style report template capturing the persistence payload, key Sysmon events, indicators, and MITRE mapping.

Example MITRE ATT&CK mapping from this lab

Case Technique Sysmon / Log Source Detection notes
Encoded PowerShell persistence T1059.001 · PowerShell Sysmon Event ID 1 + PowerShell script block (4104) Full command line and decoded script reveal suspicious encoded payload execution.
Run key persistence payload T1547.001 · Run Keys Sysmon Event ID 13 Registry value set events show the Run key path and the payload configured to launch at logon.
Command and control beaconing T1071 · Application Layer Protocol Sysmon Event ID 3 + 22 Repeated outbound connections and matching DNS queries tie the host to external C2 infrastructure.
Post exploitation host discovery T1082 · System Information Discovery Sysmon Event ID 1 Commands like systeminfo and whoami generate process events that outline the discovery phase.

This framework can be extended as more simulations are added, while staying grounded in the same Sysmon and Windows telemetry used throughout the lab.
