Lab Walkthrough

Phase 1 — Host Preparation & Baseline

The value of Sysmon depends entirely on how well you understand what's normal on the host. Privacy settings were tightened, background processes reduced, and a baseline of running processes and discovery commands was captured before any simulated attacks began. This becomes the reference point for every later investigation.

Getting the host ready

  • • Hardware and display tuned so consoles are readable side by side
  • • PowerShell process list captured as the clean baseline

Phase 3 — Event IDs & Custom Views

Raw Sysmon output is noisy — registry events alone can flood the log. Custom views were built around a small set of high-value event IDs that map directly to attacker behavior. This keeps hunting fast and focused rather than scrolling through thousands of entries.

The six event IDs that matter most

  • EID 1 — Process creation (parent, image path, full command line)
  • EID 3 — Network connections (process → remote host and port)
  • EID 11 — File creates (payload and DLL drops)
  • EID 13 — Registry value set (Run key persistence)
  • EID 22 — DNS queries (ties network activity to domains)
  • Script block — Encoded PowerShell detection

Custom view built around core Sysmon event IDs — used throughout all investigations

EID 13 — registry value set, the primary signal for Run key persistence

Phase 4 — Threat Simulation & Hunting

Attacker behavior was introduced in stages — starting with normal activity to establish what benign looks like, then escalating through LOLBin downloads, Run key persistence, beacon traffic, and encoded PowerShell. Every action is visible in Sysmon and forms part of a reconstructed story.

LOLBin downloads

Built-in tools like Invoke-WebRequest were used to pull remote files — demonstrating how attackers blend with admin-style actions. Sysmon captured both the process and the network connection.

Invoke-WebRequest download — Sysmon captures process and network connection together

Run key persistence

PowerShell was used to write a Run key entry. The change appeared immediately as EID 13, and the cleanup attempt was also visible — showing that attacker remediation leaves its own trail.

EID 13 capturing the Run key write — payload path and registry key both visible

C2 beacon traffic

Repeated outbound connections from PowerShell to a remote host simulated beacon behavior. EID 3 network events combined with EID 22 DNS queries linked the process to the remote infrastructure.

Beacon-style EID 3 activity — repeated connections to the same host and port

Encoded PowerShell

An encoded PowerShell command was executed to simulate obfuscated delivery. Script block logging captured the decoded content, and the execution timestamp tied it to registry and network events in the same session.

Script block log of an encoded command — decoded content reveals the payload intent

Phase 5 — SOC Reporting & MITRE Mapping

Each simulation was treated like a real ticket: summarize what happened, list key events and indicators, explain why it's suspicious, and tie it to MITRE ATT&CK. The goal is for someone who didn't run the lab to fully understand the incident from the report alone.

Case template structure

  • • Summary — what happened and when
  • • Key commands and PowerShell activity
  • • Sysmon event IDs that support the finding
  • • Registry paths, payload details, or network indicators
  • • MITRE technique reference and detection notes

What I Learned

Key takeaways

  • • A single well-instrumented endpoint can provide enough signal to reconstruct full attack stories
  • • Most of the value comes from a small group of Sysmon event IDs when they're filtered and interpreted correctly
  • • Registry and network telemetry are powerful for persistence and C2 detection — especially when tied to process context
  • • Encoded PowerShell still leaves clear script block entries once decoded — obfuscation doesn't hide the intent
  • • Structured SOC reporting and MITRE mapping are as important as the logs themselves for operational readiness
