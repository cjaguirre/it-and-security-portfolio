Windows Sysmon Threat Detection Lab

In this lab I worked as a blue team analyst on a single Windows host with Sysmon installed. I tuned logging, explored high value event IDs, used Event Viewer and PowerShell to hunt for attacker behavior, and wrote a short SOC style report that ties everything to MITRE ATT&CK. The screenshots below walk through the environment, the hunting workflow, and several real investigations.

Endpoint telemetry and log hunting Tools: Sysmon, Windows 10, Event Viewer, PowerShell

Workflow Overview

Step 1

Lab goal

Build a single host lab that shows how Sysmon turns Windows activity into detailed security telemetry.

Step 2

Environment setup

Prepare a Windows 10 virtual machine, tune hardware for smooth use, and harden basic privacy settings.

Step 3

Sysmon and logging

Install Sysmon with a community driven configuration and confirm that the Operational log is collecting data.

Step 4

Threat hunting

Use filters and PowerShell queries to follow processes, registry keys, network events, and files.

Step 5

Reporting

Turn the findings into a short SOC report that maps each case to MITRE ATT&CK techniques.

Lab Breakdown

1. Windows VM and lab environment

Getting the virtual machine ready for Sysmon and log analysis

I started with a dedicated Windows 10 virtual machine that acts as the monitored host. Before touching security tools, I confirmed that the VM hardware settings, display, and privacy options were in a good state for day to day analysis work. This keeps the lab stable and avoids noise from constant system changes.

These screenshots show how I confirmed the environment, from simple desktop checks to hardware display settings and privacy prompts in the operating system.

Windows 10 privacy settings during setup, reducing unwanted telemetry before security testing.

Virtual machine display settings tuned for a clear view of Event Viewer and PowerShell windows.

Final Windows host desktop that I used during the lab, with the tools I needed pinned and ready.

Key actions in this step

  • Confirmed VM resources and display so I can keep multiple windows open while hunting.
  • Walked through Windows setup screens with privacy and security in mind.
  • Captured a clean baseline view of the host before any attacker activity is reviewed.

2. Sysmon installation and configuration

Installing Sysmon with an enterprise style configuration and confirming the log

Next I installed Sysmon with administrator rights and applied a community configuration that is similar to what many blue teams use in production. The goal here is to log rich security events without drowning in noise. I also gathered notes and links from GitHub so I could explain why certain events matter.

After installation, I moved straight into Event Viewer to confirm that the Sysmon Operational log was present, healthy, and collecting events in real time.

Sysmon installation confirmed with administrator rights so the driver and service can monitor the system.

Reviewing a community Sysmon configuration from GitHub to understand which events are enabled and why.

Initial Event Viewer view before I narrow down the focus to security relevant logs.

Sysmon Operational channel selected, confirming that the engine is logging detailed activity.

Key actions in this step

  • Installed Sysmon as a persistent service that tracks process, network, registry, and file activity.
  • Used a proven configuration from the community rather than raw defaults.
  • Verified that Sysmon events are flowing into the Operational log before moving to threat hunting.

3. High value event IDs and log filtering

Finding the right events instead of scrolling through noise

Sysmon can produce a huge amount of data, so I focused on a small group of high value event IDs and then used filters to surface what matters. I started by drilling into process, registry, file, and DNS events.

I used Event Viewer filters to target specific event IDs, then built a custom view that only includes the sources and keywords that matter for investigations.

Filtering the log down to specific Sysmon event sources so my view stays focused on security data.

Selecting keywords while building a custom view so I can quickly highlight suspicious activity.

Event ID 3 network connection filter that isolates traffic to and from suspicious ports and hosts.

Event ID 13 registry value set, which is important for tracking persistence keys and malware settings.

Key actions in this step

  • Identified process, registry, file, network, and DNS events that carry the most security value.
  • Created filtered views instead of scrolling through the entire event log.
  • Captured examples of each event type that I later reference in my report and investigations.

4. Threat hunting and investigations

Following the trail from normal behavior to suspicious activity

With the groundwork in place, I moved into actual hunting. I started with simple baselines such as a normal browser launch, then introduced anything that looked suspicious, including test files and living off the land binaries for downloads.

The goal here was to practice reading Sysmon events like a story. I looked at how an action starts, what files and keys it touches, and how it might talk out to the internet or prepare persistence.

Baseline investigation of a normal browser launch so I can later compare it with suspicious runs.

Key actions in this step

  • Built a mental baseline of what normal web and file activity looks like in Sysmon.
  • Simulated common attacker techniques such as using trusted tools to download files.
  • Practiced reading registry and network events as part of a complete investigation story.

5. Attacker behavior, command and control, and reporting

From encoded commands to post exploitation and SOC ready notes

Summary

This Sysmon project shows how I work through an investigation from start to finish on a single Windows host. I prepared the virtual machine, installed and verified Sysmon, tuned Event Viewer to focus on the right event IDs, and then walked through a series of cases that cover downloads, persistence, discovery commands, command and control, and post exploitation activity. Each screenshot in this page comes from my own lab work and is tied to a concrete step or decision a defender would make. The result is a practical example of how I use Windows telemetry, PowerShell, and structured reporting to understand and explain attacker behavior.
