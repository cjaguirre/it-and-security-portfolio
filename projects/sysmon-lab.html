                Windows Sysmon Threat Detection Lab            

           

                Built a Windows Sysmon lab inspired by the TryHackMe room to practice real world threat detection.                 Deployed and verified Sysmon, tuned log views around high value event IDs, investigated simulated attacks,                 and packaged everything into a SOC style incident report mapped to MITRE ATT&CK.            

           
                                                        Endpoint telemetry · Windows logging                                                                         Skills: Sysmon, Event Viewer, PowerShell, Threat Hunting, MITRE Mapping                            
           
                               
                   

                        Executive Summary                    

                   

                        The goal of this lab is to show how a junior analyst can use Sysmon logs to detect malware,                         persistence, and command and control activity on a Windows host. I worked from the perspective of                         a SOC analyst who receives telemetry from an endpoint rather than someone who only runs tools.                    

                   

                        I prepared a Windows virtual machine, confirmed privacy and telemetry settings, verified that                         Sysmon was installed with an enterprise style configuration, and then focused on high value event                         IDs such as process creation, network connections, registry changes, and file activity. From there                         I investigated several attack scenarios and turned the results into a structured incident report                         tied to MITRE ATT&CK techniques.                    

               
                               
                   

                        What I Built                    

                   
                           
  • A Windows 10 monitoring VM prepared specifically for Sysmon and log analysis.
    •                        
  • Sysmon installed with a community configuration that mirrors enterprise deployments.
    •                        
  • Custom Event Viewer filters focused on process, network, registry, file, and DNS telemetry.
    •                        
  • PowerShell plus XPath queries to hunt through large event volumes efficiently.
    •                        
  • Four full investigations that walk through realistic attacker behavior.
    •                        
  • A SOC style report with executive summary, environment details, and case write ups.
    •                        
  • A MITRE ATT&CK mapping table that links each simulation to techniques and event IDs.
    •                    
               
           
       
   
       
       
           
               

Incident Response Workflow

                           
           
               
                   

Phase 1

                   

Environment Prep

                   

                        Prepared a Windows VM with privacy, display, and baseline tools so Sysmon activity is easy to review.                    

               
               
                   

Phase 2

                   

Sysmon Install

                   

                        Installed Sysmon and applied a tuned configuration, then confirmed events flowed into the Operational log.                    

               
               
                   

Phase 3

                   

Event Tuning

                   

                        Identified high value event IDs and created custom filtered views and queries around them.                    

               
               
                   

Phase 4

                   

Investigations

                   

                        Walked through multiple attack paths and rebuilt the story only from Sysmon telemetry.                    

               
               
                   

Phase 5

                   

Reporting

                   

                        Produced a SOC style report and MITRE mapping that shows how each behavior was detected.                    

               
           
       
   
       
       
                       
               

1. Windows VM and Lab Environment

               

                    Getting the monitored host ready for Sysmon and log analysis                

               

                    I started with a dedicated Windows 10 virtual machine that acts as the monitored host. Before adding any security tools,                     I walked through the standard setup screens, confirmed privacy and telemetry settings, and created a clean desktop layout                     with PowerShell and Event Viewer easy to reach from the taskbar. The idea is to have a stable and repeatable base image                     where security changes are the only difference.                

               

                    The screenshots below document how I prepared the environment. By capturing each stage, I can later show exactly what                     changed from a default Windows install to a machine that is ready for Sysmon based hunting.                

                               
                                       

                        Windows 10 privacy options during initial setup, keeping telemetry as limited as possible before security tooling is added.                    

               
                               
                                       

                        Display and resolution settings tuned for clear views of Event Viewer and PowerShell on the lab workstation.

                    Key actions in this phase                

               
                       
  • Confirmed Windows updates, hostname, and display settings.

  • Kept the lab machine focused on security tools rather than everyday applications.

2. Sysmon Installation and Configuration

               

                    Installing Sysmon with an enterprise style config and confirming the log                

               

                    With the VM ready, I installed Sysmon as an administrator and applied a community configuration that mirrors how blue                     teams tune Sysmon in production. The goal is to log high value security events without drowning in noise. I also gathered                     notes from the configuration file so I can explain which events are being collected and why.                

               

                    After the service was installed, I went straight into Event Viewer to confirm that the Sysmon Operational log existed                     and that new events were being written in real time. Only once the telemetry was flowing did I move on to threat hunting.                

                               
                                       

                        Installing Sysmon from an elevated PowerShell window to register the driver and service on the Windows host.                    

               
                               
                                       

                        Reviewing a community Sysmon configuration file to see which event IDs are enabled and how noise is reduced.                    

               
                               
                                       

                        Event Viewer showing the Sysmon Operational channel selected and actively logging events.                    

               
                               
                                       

                        Live event stream from Sysmon, confirming that the configuration and service are working before hunting begins.                    

               
               

                    Key actions in this phase                

               
                       
  • Installed Sysmon as a persistent service that survives reboots.
    •                    
  • Applied a tuned configuration drawn from a community GitHub project.
    •                    
  • Verified that events were flowing into the Sysmon Operational channel in real time.

3. High Value Event IDs and Log Filtering

               

                    Finding the right events instead of scrolling through noise                

               

                    Sysmon can emit thousands of events on a busy system, so the next step was to narrow the view to a small group of                     event IDs that map to attacker behavior. I focused on process, network, file, registry, and DNS activity and created                     a custom view that only shows these security focused events.                

               

                    I used Event Viewer filters for quick triage and PowerShell for more advanced hunting. The combination lets me pivot                     from a suspicious port or process back to the exact event IDs that show how the activity started and what it touched.                

                               
                                       

                        Building a custom view in Event Viewer that includes only high value Sysmon events such as 1, 3, 10, 11, 13, 15, and 22.                    

               
                               
                                       

                        Applying filters for specific event IDs and keywords so investigations stay focused on suspicious activity.                    

               
                               
                                       

                        Sysmon Event ID 1 entries showing detailed process creation information such as parent process and command line.

4. Threat Hunting and Investigations

               

                    Following the trail from normal behavior to suspicious activity                

               

                    After the groundwork was in place, I shifted to threat hunting. I started by capturing normal actions such as a                     browser launch and simple downloads so I could see what a clean baseline looks like. Then I introduced activity that                     looks more like an attack and watched how Sysmon recorded the difference.                

               

                    The goal was to practice reading Sysmon events like a story. I watched how a process started, what files and registry                     keys it touched, and how network connections linked everything together. Each of these investigations becomes a case                     in the SOC style report.                

                               
                                       

                        Capturing a clean example of a normal browser session so later malicious activity has something to be compared against.                    

               
                               
                                       

                        Running simple PowerShell commands to see how basic scripting appears in Sysmon logs before adding obfuscation.                    

               
                               
                                       

                        Using a built in tool to download a file, then checking Sysmon Event ID 3 for network details and Event ID 11 for the file write.                    

               
                               
                                       

                        Correlating process, file, and network events in Event Viewer to tell the full story of a single simulated download.                    

               
                               
                                       

                        Viewing a registry key used to hide a payload for later use as part of a persistence scenario.                    

               
                               
                                       

                        Attempting to remove the persistence key and confirming in Sysmon that the underlying registry changes were recorded.                    

               
               

                    Key actions in this phase                

               
                       
  • Built a mental baseline of what normal web and PowerShell activity looks like in Sysmon.
    •                    
  • Simulated common attack behaviors such as using trusted tools to download files and set persistence.
    •

                    Summary                

               

                    This Sysmon project walks through the full life cycle of an endpoint detection task on a single Windows host.                     Starting from a clean virtual machine, I prepared the environment for monitoring, installed and verified Sysmon,                     and focused on a small set of high value event IDs. From there I used Event Viewer and PowerShell to investigate                     simulated attacks, including file downloads, persistence mechanisms, credential access, and command and control                     traffic. The work ends with a SOC style report and MITRE ATT&CK mapping that show exactly how each behavior was                     detected. Together, these pieces demonstrate how I approach Windows telemetry, threat hunting, and incident                     documentation in a way that is ready for a security analyst role.
