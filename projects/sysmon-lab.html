Windows Sysmon Threat Detection Lab

I built a beginner-friendly Windows Sysmon lab, based on the TryHackMe room, to practice how a blue-team analyst would detect real attacker activity. I deployed and tuned Sysmon, hunted through high-value event IDs, investigated multiple attacks, and turned everything into a SOC-style report with MITRE ATT&CK mapping and portfolio-ready screenshots.

Endpoint Telemetry · SOC Reporting Tools: Sysmon, Windows 10, Event Viewer, PowerShell, MITRE ATT&CK

Project Narrative

Sysmon (System Monitor) is a Windows component from Microsoft Sysinternals that records detailed process, network, registry, and file activity. In real environments this telemetry is usually forwarded into a SIEM. In this lab I focused on working like an analyst at the endpoint level: understanding what Sysmon logs, then using those logs to answer incident-response questions.

The walkthrough moves from setup to real investigations. After bringing a Windows 10 VM online and installing Sysmon, I explored the Operational log, built custom Event Viewer filters, used PowerShell for hunting, and walked through investigations such as encoded PowerShell, LOLBin-based downloads, persistence via Run keys, and post-exploitation discovery commands. Everything ends in a structured SOC-style report with MITRE ATT&CK mapping.

1. Lab and Environment Overview

Building a Windows 10 VM and a Sysmon-ready host

The lab runs on a Windows 10 virtual machine inside VMware. I treated this VM as a typical workstation that might exist in a small enterprise. From there I adjusted basic hardware settings, walked through the Windows out-of-box setup, and then prepared the system for security monitoring.

Once Windows was ready, I confirmed network connectivity and general usability so that later attack simulations would look realistic: browser usage, file downloads, and normal background activity all show up in Sysmon and in the Windows event logs.

2. Sysmon Installation and Configuration

Turning Windows event noise into usable security telemetry

Sysmon is a Windows service and driver that survives reboots and logs rich telemetry such as process creation, network connections, module loads, file activity, and registry changes. Instead of logging everything, it relies on a configuration file that decides what to capture and what to ignore.

In this lab I used a community configuration from GitHub, reviewed the rules, and then installed Sysmon as an administrator. From there I validated that events were flowing correctly by checking both the Sysmon Operational log and process listings in PowerShell.

3. High-Value Event IDs and Log Filtering

Process, network, registry, and file telemetry for hunting

Sysmon exposes many event IDs, but a smaller set does most of the heavy lifting when hunting for attacker behavior. I focused on process creation, network connections, file creation, registry modifications, and DNS queries. These events line up nicely with common attack techniques such as payload execution, command-and-control beacons, and persistence.

Event ID Meaning Use in Lab
1 Process creation Detects malware, LOLBins, discovery commands, and payloads.
3 Network connection Identifies beacons, callbacks, and suspicious ports.
11 File create Surfaces droppers, tools, and payloads written to disk.
13 Registry value set Tracks persistence changes such as Run keys.
22 DNS query Shows suspicious domains and potential C2 infrastructure.

I used Event Viewer to explore the Sysmon Operational log, then layered on filters to cut down on noise and spotlight the events that mattered. This made it much easier to pivot from “something looks off” to “this exact process on this host contacted this address over this port.”

4. Threat Hunting Techniques

Beaconing, discovery, persistence, and file-based activity

With a clean logging pipeline, I shifted into threat-hunting patterns that a SOC analyst would use in day-to-day work. I combined Event Viewer searches with PowerShell and XPath filters to move between network events, process trees, registry keys, and file creations.

Process and account discovery

  • Tracked Event ID 1 entries for discovery commands like whoami.
  • Used PowerShell to list processes and correlate them with Sysmon events.
  • Built a quick mental model of what “normal” looks like on this host.

Network and DNS-based hunting

  • Focused on Event ID 3 and Event ID 22 to spot unusual destinations and ports.
  • Matched network activity back to the originating processes and command lines.
  • Looked for long-lived connections that could represent beacons or C2 channels.

Registry and persistence hunting

  • Monitored registry events, especially Event ID 13, for suspicious Run keys.
  • Flagged values that launched PowerShell or unknown binaries at startup.

File creation and tool detection

  • Used Event ID 11 to watch for new executables and DLLs written to disk.
  • Correlated file creation with network and registry events to build a full story.

5. Practical Investigations

From single events to full attack stories

The core of the lab is a series of investigations that mimic real SOC questions: “What launched this PowerShell command?”, “How did this payload arrive?”, or “Where is persistence hiding?”. Below are examples of how Sysmon events, filters, and PowerShell all come together to answer those questions.

Case 1 — Encoded PowerShell and script block logging

Goal: understand how an attacker used encoded PowerShell and confirm what the script actually did.

Case 2 — LOLBin-based payload downloads

Goal: follow how a seemingly harmless Windows binary was used to download a payload.

Case 3 — Registry Run key persistence

Goal: identify and validate persistence set up via Run keys in the registry.

Case 4 — Post-exploitation discovery

Goal: show how simple discovery commands can still be tied back to the broader intrusion.

6. SOC-Style Reporting and MITRE ATT&CK Mapping

Packaging investigations into something a SOC could send to leadership

The last step was to turn the raw notes, screenshots, and events into a clean report. I wrote an executive summary in plain language, documented the lab environment, listed out the most important Sysmon event IDs, and then broke each case into its own mini-story with timeline, findings, and recommendations.

Report structure

  • Executive summary focused on the overall attack flow and detection coverage.
  • Environment section that explains the Windows 10 VM, Sysmon setup, and log sources.
  • Sysmon event breakdown highlighting which IDs were most useful in each case.
  • Per-case analysis with commands, event IDs, indicators, and analyst commentary.

MITRE ATT&CK mapping

  • Encoded PowerShell → T1059.001 (Command and Scripting Interpreter: PowerShell).
  • Registry Run key persistence → T1547.001 (Boot or Logon Autostart Execution).
  • LOLBin-style download → T1105 (Ingress Tool Transfer) using built-in tools.
  • Account and system discovery → T1033 / T1082 via whoami and systeminfo.

The result is a complete story: from turning on a Windows VM, to installing Sysmon, to catching encoded PowerShell and persistence, and finally to presenting everything in a format that security teams and hiring managers can quickly understand.
