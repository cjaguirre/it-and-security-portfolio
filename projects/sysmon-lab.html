Windows Sysmon Threat Detection Lab

Built a Windows Sysmon lab based on a guided training room to practice detection skills on a single host. Prepared a clean Windows monitoring machine, installed and tuned Sysmon, hunted through key event IDs, walked through several investigations, and wrapped everything into a SOC style report mapped to MITRE ATT&CK.

Endpoint telemetry · Windows logging Tools: Sysmon, Event Viewer, PowerShell, MITRE ATT&CK, SOC reporting

Project Narrative

I treated this lab as if I were a junior analyst handed a single Windows endpoint to monitor. The focus was not only on running attacks but on reading Sysmon telemetry as a story. I started from the very beginning: a fresh Windows host, privacy and display tuned, then Sysmon installed with a community configuration that matches how blue teams use it in real environments.

Once the logging pipeline was confirmed, I narrowed in on the event IDs that matter most for detection and created filtered views to keep the noise down. From there I walked through investigations covering file downloads, persistence, credential access, and command and control traffic. The end result is a complete SOC style write up that shows what happened, how it was detected, and which ATT&CK techniques are involved.

What I Built

  • A Windows 10 monitoring VM with privacy, display, and tool layout tuned for log analysis.
  • Sysmon installed with an enterprise style configuration plus verification that events are flowing.
  • Custom Event Viewer filters and PowerShell plus XPath queries for targeted hunting.
  • A breakdown of high value Sysmon event IDs for process, network, file, registry, and DNS activity.
  • Multiple investigations that rebuild attacker behavior using only Sysmon telemetry.
  • A SOC style report structure with executive summary, environment, and individual cases.
  • A small MITRE ATT&CK mapping table that links each simulation to techniques and event IDs.

3. High Value Event IDs and Log Filtering

Focusing on process, network, file, registry, and DNS events that matter most

Once Sysmon was generating data, I tuned my views so that hunting sessions stayed focused on high value telemetry. Rather than scrolling through every event, I built a custom view anchored around a small set of event IDs that align with common attack techniques.

The first screenshot here shows the creation of a custom view in Event Viewer. I selected Sysmon as the source and then specified only the event IDs that I planned to use throughout the lab. This turns Event Viewer into a security focused dashboard instead of a general purpose log browser.

Building a custom view that tracks a small set of high value Sysmon event IDs used throughout the lab.

Next, I built a filter specifically for network events. The screenshot below shows a view restricted to Event ID 3, which logs outbound and inbound connections. Focusing on this ID makes it easier to spot beacon patterns, unusual ports, or connections tied to persistence mechanisms later on.

Example filter that isolates Event ID 3 network connection events, ready for hunting on specific ports or IPs.

To understand why filtering is necessary, I also captured a view that shows how noisy some IDs can be. The screenshot below highlights a high volume of Event ID 13 entries, which record registry value changes. Without filtering and context, this amount of data would quickly overwhelm an analyst.

High volume of Event ID 13 registry value set events, showing why tuned filtering is important for investigations.

I then walked through examples of each core event type. The process creation screenshot shows how Sysmon records not only the new process, but also the parent, full image path, and command line. These fields are critical for reconstructing the story behind an attack chain.

Sysmon Event ID 1 process creation, including parent process, image path, and full command line for context.

The next example focuses on Event ID 11, which tracks file creations. Here, I highlighted a DLL being written to disk. In a real case, changes like this could indicate a dropper, side loaded component, or payload for later execution.

Sysmon Event ID 11 file create event that records a suspicious DLL being written to disk as part of a test.

I then returned to Event ID 13, but this time zoomed in on a single registry value set that is directly related to persistence. This screenshot is used later in the lab when I build out a full Run key case from the logs.

Sysmon Event ID 13 registry value set, later used to track Run key persistence attempts in investigations.

Finally, I examined Event ID 22, which logs DNS queries. The screenshot below shows the properties for a single lookup. Combined with Event ID 3, these entries help link host processes to the remote infrastructure they contact, which is essential for identifying command and control.

Sysmon Event ID 22 DNS query properties, showing the domains contacted by processes during simulated beacons.

Key actions in this step

  • Built a custom Event Viewer view centered on a small set of high value Sysmon event IDs.
  • Demonstrated why raw registry and network logs need filtering before analysis.
  • Captured detailed examples of process, file, registry, and DNS events for later case work.

4. Threat Hunting and Attack Simulation

Following the trail from normal activity to suspicious behavior

With telemetry tuned, I moved into active hunting. This part of the lab introduces realistic behavior in phases, starting with harmless actions, then escalating to LOLBin based downloads, registry persistence, and command and control style traffic. Each screenshot represents a moment in the story that I later convert into a SOC case.

To establish a baseline, I began with a simple web browsing session. The screenshot below shows a normal browser launch. The point here is not the website itself, but rather how standard user activity appears in Sysmon so that abnormal network patterns will stand out later.

Baseline investigation of a normal browser launch to see what clean, non malicious web usage looks like in Sysmon.

I then created a simple test file and watched Sysmon record the file creation event. By starting with a harmless file, I can later compare how malicious payload drops or DLL downloads differ in path, naming, and parent process relationships.

Simple test file creation that produces a Sysmon file create event, used as a reference before any malware style activity.

After baselining, I moved into LOLBin based downloads. The next screenshot shows the use of a curl style command from PowerShell to pull a file from a remote server. This is a common attacker pattern because it blends into administrative behavior while still retrieving an external payload.

Using a trusted tool with a curl style command to download a file, then validating that Sysmon captures the process and network details.

I repeated the download using Invoke-WebRequest to show how a similar technique can appear with different command lines. Comparing these two screenshots side by side with the logs teaches how a defender can build signatures that focus on behavior rather than a single exact string.

Alternate download test using Invoke WebRequest so I can compare how different commands appear in Sysmon logs.

Next, I focused on persistence through the registry. The first screenshot in this mini sequence shows a PowerShell command that sets a Run key value by modifying the registry with Set-ItemProperty. This operation is later recorded in Sysmon as Event ID 13 and becomes one of the key indicators in the persistence case.

PowerShell command used to create or update the Run key value, later captured in Event ID 13 registry events.

The second screenshot shows the registry based Run key entry as seen from a defender perspective. Here the value points to the payload that will execute at logon. This view is much closer to what an analyst would see during a real response effort.

View inside Event Viewer showing how the new Run key entry appears from a defender perspective.

To test detection of cleanup attempts, I tried removing the persistence key and watched how Sysmon recorded the change. The screenshot below captures that removal attempt. In a real case, this type of log can reveal where an attacker tried to cover their tracks but left partial artifacts behind.

Attempted removal of the persistence key and the matching registry events that show the change being recorded.

I then pivoted back into Sysmon to view the registry value set that represents this persistence in more detail. The next screenshot highlights the exact Event ID 13 record that ties the Run key path to the payload. This is the log entry I reference later in my written case summary.

Registry value set observed in Sysmon that stores a payload path, forming the basis of a Run key persistence case.

After persistence, I moved into simulated command and control traffic. The next screenshot shows a Sysmon Event ID 3 entry that records a network connection to a remote host. This becomes the network pivot point that links registry changes and encoded commands to external infrastructure.

Sysmon Event ID 3 network connection that links PowerShell activity to an external command and control server.

I then zoomed in on a separate network event representing a beacon style connection. Here the focus is on the repeated communication to the same host and port, which is characteristic of many C2 frameworks and botnets.

Investigation of Sysmon Event ID 3 entries that show a beacon style network connection to a command and control server.

To tie everything together, I executed an encoded PowerShell command. The next screenshot shows the script block view of that payload. From here I can decode the command, identify the intent, and match it to the registry changes and network traffic captured earlier in the lab.

Script block view of an encoded PowerShell command that is later tied to persistence and network activity.

The final screenshot in this section captures the moment that encoded command executes. This gives me the exact process details and timing needed to correlate the event with registry writes and network connections, forming a complete attack storyline from a single host.

Execution attempt for the encoded command, including process context that will be referenced in the SOC report.

Key actions in this step

  • Established normal browsing and file activity as a baseline for comparison.
  • Simulated attacker behavior with LOLBins, registry persistence, and beacon style network traffic.
  • Captured encoded PowerShell usage that ties process, registry, and network events into a single narrative.

5. SOC Reporting and MITRE ATT&CK Mapping

Turning Sysmon telemetry into a structured incident report

The last stage of the lab is reporting. Here I treat each simulation as if it were a real ticket and document the key findings for a senior analyst. That includes the commands used, the Sysmon event IDs that revealed the activity, and a short explanation of why the behavior is suspicious. I then tie these findings back to MITRE ATT&CK techniques.

The first screenshot in this section shows a post exploitation view of whoami usage. Seeing the same command appear again at this phase helps emphasize how attackers chain simple discovery with more advanced techniques, and why even basic events can matter when placed in context.

Post exploitation use of whoami, revisited as part of a larger sequence of discovery and persistence steps.

The next screenshot captures systeminfo being run after the host is already under attacker control. Together with the process creation logs from earlier, this reinforces how Sysmon can reconstruct the full discovery phase even when the attacker uses native tools.

Post exploitation use of systeminfo to pull host details, recorded as additional process creation events.

Finally, I pulled everything into a SOC style reporting template. The screenshot below shows one of the cases built around persistence. It documents the registry path, payload, supporting Sysmon event IDs, and mapped MITRE techniques. This is the type of artifact that can be attached directly to a ticket or used as a training example for other analysts.

SOC style report template capturing the persistence payload, key events, indicators, and mapped MITRE techniques.

From here it is straightforward to extend the report with more simulations or to adapt the template to other hosts. The combination of structured narrative, screenshots, and event references makes it easy for someone who did not run the lab to still understand what happened and why it matters.

Example MITRE ATT&CK mapping used in this lab

Case Technique Sysmon / Log Source Detection notes
Encoded PowerShell persistence T1059.001 · PowerShell Sysmon Event ID 1 + script block log (4104) Full command line and decoded script reveal suspicious payload execution.
Run key payload in registry T1547.001 · Run keys Sysmon Event ID 13 Registry value set events show where the payload is stored and which key launches it.
Command and control beacon T1071 · Application layer protocol Sysmon Event ID 3 and 22 Outbound network connections and DNS queries tie the beacon to its remote infrastructure.
Post exploitation host discovery T1082 · System information Sysmon Event ID 1 Commands like systeminfo and whoami generate process events that reveal discovery steps.

This mapping framework can be extended with additional techniques as more simulations are added, while still staying grounded in the same Sysmon telemetry that powers the rest of the lab.
