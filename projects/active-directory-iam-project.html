Active Directory Lab Windows Server 2022 Microsoft Azure VM Identity and Access Management Helpdesk Workflow

Active Directory Beginner Lab (Azure-hosted)

Deployed a Windows Server 2022 virtual machine in Microsoft Azure, promoted it to a domain controller, and walked through the full identity lifecycle for a new user. Created accounts, groups, and organizational units, managed passwords and lockouts, and configured a shared folder to show how everyday access requests are handled in a domain-backed environment.

Entry level IAM and helpdesk workflow, cloud-hosted Tools: Azure VM, Windows Server 2022, AD DS, DNS, ADUC, RDP

Project Narrative

This lab begins in Microsoft Azure, where I provisioned a Windows Server 2022 Datacenter virtual machine and prepared it to act as a domain controller. From there I used Server Manager, SCONFIG, and the Add Roles and Features wizard to install Active Directory Domain Services (AD DS) and DNS, then promoted the server into a new forest and domain for the lab.

After the promotion and reboot, the Azure VM functioned as DC01 for the adlab.local domain. I created a dedicated domain administrator account, then used Active Directory Users and Computers (ADUC) to build out a realistic identity workflow: creating a test user, enforcing a first-logon password change, and managing group membership to control access.

The final part of the lab focused on access control. I created a security group, organized it into a custom OU, and wired it to a shared folder on the domain controller. By resetting passwords, unlocking accounts, and adding or removing users from groups, I simulated the day-to-day identity tickets that entry-level IT, helpdesk, and IAM analysts handle in a Windows domain backed by Active Directory.

Lab breakdown

1. Azure VM deployment and baseline server configuration

Deployed a Windows Server 2022 VM in Azure and prepared it to act as a domain controller.

The lab starts in the Azure portal by creating a new Windows Server 2022 virtual machine. I selected a region close to my location, chose an appropriate VM size, and enabled RDP access so I could administer the machine remotely. Once the VM booted, I connected over Remote Desktop and landed in Server Manager, which acts as the main control plane for Windows Server roles and features.

From there I opened the SCONFIG utility to perform initial hardening and identity work. I confirmed basic network configuration and renamed the computer to DC01, then restarted the server to lock in the new hostname. This mirrors what system administrators do when preparing a dedicated host to carry domain services.

2. Installing Active Directory Domain Services and DNS

Added the directory and name services that support sign-in and domain identity.

With the base Azure server ready, I used the Add Roles and Features wizard in Server Manager to install Active Directory Domain Services and DNS. These two roles are the core of a Windows domain: AD DS handles authentication and identity, while DNS resolves hostnames and supports domain operations.

  • Launched the Server Manager roles wizard and targeted the DC01 server instance.
  • Enabled Active Directory Domain Services and accepted all required management features.
  • Added the DNS Server role so DC01 could provide name resolution for the lab domain.
  • Completed installation and prepared to promote DC01 into a domain controller role.

3. Promoting DC01 and creating the adlab.local domain

Turned the Azure VM into a domain controller and created a dedicated forest for the lab.

Once AD DS was installed, Server Manager displayed a notification to promote the server to a domain controller. I launched the configuration wizard, chose to add a new forest, and created the adlab.local domain that serves as the identity backbone for the entire lab. During this process I set the Directory Services Restore Mode (DSRM) password and confirmed DNS integration.

  • Selected the option to add a new forest and specified adlab.local as the root domain name.
  • Confirmed that DC01 would act as both a domain controller and DNS server.
  • Reviewed database, log, and SYSVOL paths for Active Directory storage.
  • Completed the wizard and allowed the Azure VM to restart as a fully functional domain controller.

4. User lifecycle and group management in ADUC

Practiced the daily tasks a helpdesk or IAM analyst uses to manage accounts.

After promoting the domain controller, I signed in using a dedicated domain administrator account (dcadmin) and opened Active Directory Users and Computers. I enabled Advanced Features to expose additional containers and attributes, then walked through the full lifecycle of a typical user, including onboarding, password policy, and troubleshooting.

  • Created a user named Test User with a logon name of test.user in the adlab.local domain.
  • Required the user to change their password at the next logon to mirror real onboarding practices.
  • Attempted sign-in as test.user from the RDP session to validate the account and password policy.
  • Created a security group named AI to represent a functional team or application access role.
  • Added the Test User account to the AI group and later removed it to simulate routine access changes.
  • Performed a password reset as dcadmin and used the “Unlock account” option to resolve simulated lockouts.

The snapshot below shows the Test User account and the AI security group inside ADUC. These objects drive most of the identity lifecycle work in the lab: onboarding, membership changes, and support tickets.

5. Organizational units, nested groups, and shared resources

Organized directory objects and connected group membership to a real shared folder.

To design a directory structure that looks closer to production, I created a dedicated Organizational Unit (OU) for lab groups and moved the AI group into it. I then tied that group to a shared folder on DC01 so that membership in Active Directory directly controlled access to real resources.

  • Created an OU named Custom Groups with protection from accidental deletion enabled.
  • Moved the AI security group into the Custom Groups OU to separate lab objects from built-in containers.
  • Created a shared folder named corp_shared on DC01 and exposed it over the network.
  • Granted access to corp_shared by assigning permissions to the AI group instead of individual users.
  • Verified that adding or removing the Test User from the AI group immediately affected access to \\DC01\corp_shared.

The screenshot below highlights how the Custom Groups OU and the shared folder permissions align. This combination demonstrates how clean OU design plus group-based access control makes permissions easier to reason about and maintain over time.

Sample administrative commands

While most of the lab work was done through Server Manager and ADUC, I can manage the same identity lifecycle using PowerShell once the domain is in place. These sample commands show how to automate user creation, group membership, and password resets in the adlab.local environment hosted on Azure.

Create and enable a new user 

New-ADUser -Name "Test User" `
           -GivenName "Test" `
           -Surname "User" `
           -SamAccountName "test.user" `
           -UserPrincipalName "test.user@adlab.local" `
           -AccountPassword (Read-Host -AsSecureString "Password") `
           -ChangePasswordAtLogon $true `
           -Enabled $true

Group membership and password reset 

# Add user to AI group
Add-ADGroupMember -Identity "AI" -Members "test.user"

# Reset password and require change at next sign in
Set-ADAccountPassword -Identity "test.user" `
                      -Reset `
                      -NewPassword (Read-Host -AsSecureString "NewPassword")
Set-ADUser -Identity "test.user" -ChangePasswordAtLogon $true

Architecture overview

Components

  • Azure Virtual Machine — Windows Server 2022 Datacenter instance hosting the domain controller.
  • Domain controller DC01 — Runs AD DS and DNS for the adlab.local forest.
  • Active Directory database — Stores users, groups, computers, and organizational units.
  • DNS service — Resolves host names and supports domain operations for the lab environment.
  • Shared folder corp_shared — File share secured by the AI group and used to test access control.
  • RDP client session — Represents the user or admin workstation connecting into the environment.

Identity and access flow

  • A user signs in over RDP using credentials for an account stored in Active Directory.
  • The domain controller DC01 validates the password and evaluates the user’s group membership.
  • Group membership places the account in AI and any other lab groups that carry permissions.
  • When the user connects to \\DC01\corp_shared, Windows checks the user’s group tokens against the share and NTFS ACLs.
  • Changes to membership, password resets, or lockout/disable actions in AD immediately change access to the share.

Key insights

  • Active Directory remains the central source of truth for users, groups, and access across Windows environments, whether the domain controller runs on-premises or in Azure.
  • Most day-to-day tickets for junior analysts revolve around a small set of high-impact tasks: creating accounts, resetting passwords, unlocking users, and adjusting group membership.
  • Organizational units and clean group design make permissions easier to reason about, reduce human error, and support future automation.
  • Managing access through security groups tied to shared folders scales much better than assigning rights directly to individual users.
  • Hosting the lab in Azure provides a safe, disposable environment to practice real enterprise workflows without impacting any production systems.

This Active Directory lab walks through the full identity lifecycle that supports a small Windows domain, hosted safely on a Windows Server 2022 virtual machine in Microsoft Azure.
