Project narrative

I deployed a Windows Server 2022 VM in Azure and configured it as DC01, the first domain controller of a new forest named adlab.local. After promoting the server, I used Active Directory Users and Computers (ADUC) to create users, organize objects, and validate that core services like AD DS and DNS were working end to end.

The lab focuses on the identity lifecycle: onboarding a user, assigning group membership, applying permissions to a shared resource, and performing routine support work such as password resets and account updates. It mirrors the type of troubleshooting and change requests handled by entry level IT and identity analysts.

What I built

  • Deployed a Windows Server 2022 VM in Azure and configured it as DC01.
  • Installed AD DS and DNS and promoted the server to the adlab.local domain.
  • Created a domain admin account for clean separation of admin tasks.
  • Designed an initial OU structure and created users and security groups in ADUC.
  • Configured a shared folder on DC01 and secured it using group-based permissions.
  • Practiced help desk style tasks such as password reset, account unlock, and group changes.

Lab breakdown

1. Azure VM deployment and baseline server configuration

Set up the Windows Server 2022 VM that becomes the domain controller

The lab starts with an Azure Virtual Machine running Windows Server 2022. I chose the region and size, enabled RDP access, and named the host DC01 so it was ready to be promoted later as a domain controller.

Azure VM creation settings for the Windows Server 2022 host, including region, size, OS image, and baseline configuration.

After deployment I connected with RDP, used Server Manager and SCONFIG to verify networking, renamed the server, and restarted it to apply the changes. At this point DC01 was stable and ready for domain services.

Server Manager dashboard after first login, showing DC01 ready for AD DS and DNS installation.

2. Installing Active Directory Domain Services and DNS

Added the core directory and name services required for authentication

With DC01 prepared, I opened the Add Roles and Features wizard and installed Active Directory Domain Services (AD DS) and DNS Server. AD DS provides the identity database and authentication logic, while DNS supports domain name resolution and allows clients to locate controllers and services.

Add Roles and Features wizard configured to install AD DS and DNS, the foundation of the domain.

After the roles installed, Server Manager surfaced a notification to promote DC01 to a domain controller. That next step shifts the server from a standalone host into the identity backbone for the lab environment.

Installation results confirming that AD DS and DNS are in place and ready for promotion.

3. Promoting DC01 and creating the adlab.local domain

Turned the Azure VM into a domain controller and built the new forest

From the Server Manager notification I launched the Active Directory Domain Services Configuration Wizard, selected the option to create a new forest, and specified adlab.local as the root domain. This step defined the logical boundary for all users, groups, and computers in the lab.

Domain Services configuration wizard used to create the new adlab.local forest.

Before continuing, Windows ran a prerequisites check to validate DNS configuration, credentials, and security settings. Only after these checks passed was DC01 allowed to become a domain controller.

Prerequisite checks confirming DC01 is ready to take on domain controller responsibilities.

After a reboot, DC01 came back as a fully functional domain controller. Using Active Directory Users and Computers, I verified that the new domain and default containers were present, confirming that the forest was healthy and ready for user and group creation.

ADUC view after promotion, showing the newly created adlab.local domain structure.

4. User lifecycle and group management in ADUC

Practiced the daily identity tasks handled by support and IAM teams

With the domain online I opened ADUC and walked through the lifecycle of a sample user. I created a Test User, assigned an initial password, and required the password to be changed at first sign in, just as a real onboarding process would enforce.

User creation in ADUC, including password configuration and first sign in change requirements.

I also created a security group named All to represent a team or role. Together the Test User and the All group formed the core identity objects used throughout the lab for testing group based permissions and access changes.

ADUC view showing the Test User and All security group used for role based access testing.

5. OUs, security groups, and shared folder access

Organized directory objects and applied group permissions to a shared resource

The final part of the lab focused on access control design. I created a dedicated Custom Groups OU, moved the All group into it, and built a shared folder on DC01 named corp_shared. Rather than assigning permissions directly to users, access was granted to the All group.

This approach reflects how real environments scale: user membership changes while group permissions stay stable. Adding the Test User to All granted access to corp_shared immediately, and removing the account removed access just as quickly.

  • Created a Custom Groups OU with accidental deletion protection enabled.
  • Moved the All security group into the new OU for better organization.
  • Created the corp_shared folder and published it as a network share.
  • Assigned share and NTFS permissions to the All group instead of individual users.
  • Verified that group membership directly controlled access to the shared folder.

Shared folder permissions assigned through the All security group, demonstrating clean group based access control.

Sample administrative commands

Most of the lab uses graphical tools like Server Manager and ADUC, but the same identity lifecycle can be automated through PowerShell once the domain is in place. These sample commands show how to handle core tasks such as account creation, group membership, and password reset in the adlab.local environment.

Create and enable a new user 

New-ADUser -Name "Test User" `
           -GivenName "Test" `
           -Surname "User" `
           -SamAccountName "test.user" `
           -UserPrincipalName "test.user@adlab.local" `
           -AccountPassword (Read-Host -AsSecureString "Password") `
           -ChangePasswordAtLogon $true `
           -Enabled $true

Group membership and password reset 

# Add user to All group
Add-ADGroupMember -Identity "All" -Members "test.user"

# Reset password and require change at next sign in
Set-ADAccountPassword -Identity "test.user" `
                      -Reset `
                      -NewPassword (Read-Host -AsSecureString "NewPassword")
Set-ADUser -Identity "test.user" -ChangePasswordAtLogon $true

Key insights

  • Active Directory is the central identity and access platform for Windows based environments.
  • Many support tickets center on user onboarding, password resets, account unlocks, and fixing group membership.
  • Clean OU design and group based permissions make access easier to manage and reduce configuration mistakes.
  • Assigning permissions to security groups instead of individuals keeps the environment scalable and easier to audit.
  • Running this lab in Azure provides a safe place to practice directory operations without touching production systems.

Summary

This lab shows how identity is created, managed, and secured inside a Windows domain. Using a Windows Server 2022 VM in Azure, I promoted DC01 to a domain controller, built the adlab.local forest, created users and groups, and organized them into a simple but clean OU structure.

By applying group based permissions to a shared folder and practicing tasks such as password reset and membership changes, the exercise mirrors the daily responsibilities of entry level IT and identity analysts. It connects the theory of authentication and authorization with the practical tools used to keep access aligned with business needs.
