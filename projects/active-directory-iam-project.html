Active Directory Beginner Lab (Azure-hosted)

Built a Windows Server 2022 domain controller in Azure and walked through the full user lifecycle. Created accounts, groups, organized OUs, managed passwords, and configured a shared folder to demonstrate real access control workflows.

SOC / IAM Workflow · Cloud-Hosted AD Lab Tools Used: Azure VM, Windows Server 2022, AD DS, DNS, ADUC, RDP

Project Narrative

I deployed a Windows Server 2022 VM in Azure and configured it as a domain controller for a new forest, adlab.local. After promoting the server to a DC, I used Active Directory Users and Computers (ADUC) to create accounts, organize OUs, and walk through the full identity lifecycle.

To demonstrate permissions and access control, I built a security group, assigned it to a shared folder, and tested typical helpdesk tasks like password resets, account unlocks, and group membership updates—mirroring real junior IAM and IT workflows.

What I Built

  • Deployed a Windows Server 2022 VM in Azure and configured it as DC01.
  • Installed AD DS + DNS and promoted the server to the adlab.local domain.
  • Created a domain admin account and set up user & group management in ADUC.
  • Built a security group and custom OU structure for organization.
  • Configured a shared folder with permissions assigned through domain groups.
  • Practiced common helpdesk tasks (password reset, unlocks, group changes).

Lab breakdown

1. Azure VM deployment and baseline server configuration

Setting up the Windows Server 2022 VM that becomes the lab’s domain controller.

The lab begins in the Azure portal with the deployment of a Windows Server 2022 virtual machine. I selected the region, VM size, enabled RDP, and assigned the machine name DC01 to establish its identity from the start. This forms the foundation for promoting the server into a domain controller later in the workflow.

Azure VM creation settings for the lab server, including the Windows Server 2022 image, the DC01 hostname, and the base configuration used to host the future domain controller.

After the VM deployed, I connected via RDP and used Server Manager and SCONFIG to apply baseline configuration: confirming networking, renaming the machine to DC01, and restarting to finalize the identity. With the server prepared, it was ready for AD DS and DNS role installation.

Server Manager dashboard after the first RDP login, showing a clean Windows Server 2022 installation ready to have domain services added.

2. Installing Active Directory Domain Services and DNS

Adding the core directory and name services that power authentication and domain identity.

With the server prepared, I opened the Add Roles and Features tool in Server Manager to install the components required for a functional domain: Active Directory Domain Services and DNS Server. AD DS provides identity and authentication, while DNS enables name resolution and domain operations. The wizard begins by outlining configuration requirements before role installation.

Initial Add Roles and Features screen, summarizing prerequisites for AD DS and DNS such as proper network configuration and secure administrator credentials before installation.

After selecting the roles, Server Manager completed the installation and displayed the next required action: promoting the server to a domain controller. This step transitions the VM from a standalone Windows server into the core identity provider for the lab environment.

Role installation results showing AD DS and DNS added successfully, along with the prompt to Promote this server to a domain controller, which kicks off domain configuration.

3. Promoting DC01 and creating the adlab.local domain

Turning the Azure VM into a domain controller and creating a dedicated forest for the lab.

From the AD DS post-installation task, I launched the Active Directory Domain Services Configuration Wizard, chose to add a new forest, and created the adlab.local domain. This is the point where DC01 is defined as the first domain controller in the environment and becomes responsible for authentication.

Configuring the AD DS deployment to Add a new forest and defining the root domain name adlab.local for the lab.

Before the configuration is committed, the wizard runs a prerequisite check. This validation ensures the server can safely host the domain controller role, verifying DNS, credentials, and key AD settings.

The AD DS prerequisites check completing successfully, confirming DC01 is ready to become the domain controller for the new adlab.local forest.

After the required reboot, the Azure VM was a fully functional domain controller. I signed in and verified the new domain structure using Active Directory Users and Computers (ADUC), confirming that DC01 was now managing the adlab.local environment.

Initial ADUC view after promotion, showing the adlab.local domain and confirming that DC01 is now operating as the domain controller.

4. User lifecycle and group management in ADUC

Practicing the daily tasks a helpdesk or IAM analyst uses to manage accounts.

With the domain online, I used ADUC to walk through the lifecycle of a typical user account. I created a test user to represent a new hire and enforced the standard security policy requiring a password change at first logon, mirroring real onboarding requirements in many enterprises.

New user creation in ADUC, including the password configuration step with User must change password at next logon enabled to model secure onboarding.

I then created a security group named AI to represent a team or role in the organization. Together, the Test User account and the AI group form the core identity objects used for all subsequent access, troubleshooting, and group-based permission testing in the lab.

ADUC view showing the created Test User and the AI security group, which are used to simulate common helpdesk and IAM operations throughout the lab.

5. Organizational units, nested groups, and shared resources

Organizing directory objects and connecting group membership to a real shared folder.

The final part of the lab focused on group-based access control. I organized custom security groups into a dedicated OU, then created a shared folder named corp_shared on DC01 and tied access to the AI security group. This setup ensures that membership in Active Directory directly controls access to the resource.

  • Created an OU named Custom Groups with protection from accidental deletion enabled.
  • Moved the AI security group into the Custom Groups OU to separate lab objects from built-in containers.
  • Created a shared folder named corp_shared on DC01 and published it as a network share.
  • Granted access to corp_shared by assigning permissions to the AI group instead of individual users.
  • Verified that adding or removing the Test User from the AI group immediately affected access to \\DC01\corp_shared.

Configuring the shared folder’s permissions by selecting the AI security group from adlab.local, demonstrating clean group-based access control for the corp_shared resource.

Sample administrative commands

While most of the lab work was done through Server Manager and ADUC, the same identity lifecycle can be managed using PowerShell once the domain is in place. These sample commands show how to automate user creation, group membership, and password resets in the adlab.local environment hosted on Azure.

Create and enable a new user 

New-ADUser -Name "Test User" `
           -GivenName "Test" `
           -Surname "User" `
           -SamAccountName "test.user" `
           -UserPrincipalName "test.user@adlab.local" `
           -AccountPassword (Read-Host -AsSecureString "Password") `
           -ChangePasswordAtLogon $true `
           -Enabled $true

Group membership and password reset 

# Add user to AI group
Add-ADGroupMember -Identity "AI" -Members "test.user"

# Reset password and require change at next sign in
Set-ADAccountPassword -Identity "test.user" `
                      -Reset `
                      -NewPassword (Read-Host -AsSecureString "NewPassword")
Set-ADUser -Identity "test.user" -ChangePasswordAtLogon $true

Key insights

  • Active Directory is the core identity system for authentication and access control in Windows environments.
  • Most helpdesk and IAM tasks revolve around user onboarding, password resets, unlocks, and group membership fixes.
  • Clean OU structure and group-based permissions make access easier to manage and reduce mistakes.
  • Using security groups instead of assigning permissions individually keeps environments cleaner and scalable.
  • Running this lab in Azure provides a safe place to practice real identity workflows without impacting production systems.

Together, these steps demonstrate how authentication, authorization, and directory structure work in practice—mirroring the daily responsibilities of entry-level IT, helpdesk, and identity analysts, including password resets, account management, and access control through AD group membership.
