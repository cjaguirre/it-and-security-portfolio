Active Directory Beginner Lab

Built a small on premise Active Directory environment on Windows Server 2022, promoted it to a domain controller, and walked through the full identity lifecycle for a new user. Created accounts, groups, and organizational units, managed passwords and lockouts, and configured a shared folder to show how everyday access requests are handled.

Entry level IAM and helpdesk workflow Tools: Windows Server 2022, AD DS, DNS, ADUC

Project Narrative

This lab starts with a blank Windows Server 2022 installation and turns it into a working domain controller. I used Server Manager and the built in configuration tools to prepare the host, install Active Directory Domain Services and DNS, and create a new internal forest that acts as the lab domain.

Once the domain controller was online, I used Active Directory Users and Computers to create a test user, manage group membership, and organize objects into an appropriate structure. The workflow mirrors what a junior helpdesk or identity analyst would do when onboarding a new employee and handling day to day requests.

The final steps focus on access control. I created security groups, nested them, and tied them to a shared folder on the server. By resetting passwords, unlocking accounts, and granting or removing access, the lab shows how core identity and access management concepts play out in a real Active Directory environment.

Lab breakdown

1. Windows Server setup and baseline configuration

Built a clean Windows Server 2022 installation and prepared it to act as a domain controller.

The lab starts with a fresh Windows Server 2022 virtual machine. From Server Manager and the built in SCONFIG tool I confirmed basic settings, renamed the host to DC01, and restarted to lock in the new identity. This step reflects what an admin would do when preparing a dedicated host to carry domain services for a small environment.

2. Installing Active Directory Domain Services and DNS

Added the directory and name services that support sign in and domain identity.

With the base server ready, I used the Add Roles and Features wizard in Server Manager to add Active Directory Domain Services and DNS. These two roles work together to authenticate users, track computers in the environment, and resolve names inside the lab domain.

  • Launched the Server Manager role wizard and selected the DC01 host.
  • Enabled Active Directory Domain Services and accepted the required features.
  • Added the DNS role so the domain controller could also handle internal name resolution.
  • Completed installation and prepared to promote the server into a domain controller role.

The screenshot below captures the role selection screen. It documents the exact configuration used to transform a plain Windows Server instance into an environment that can host a full Active Directory domain.

Roles and Features wizard showing Active Directory Domain Services and DNS selected for DC01.

3. Promoting DC01 and creating the lab domain

Turned the server into a domain controller and created a dedicated forest for the lab.

After the roles finished installing, Server Manager surfaced a notification to promote the server to a domain controller. I followed the configuration wizard, chose to add a new forest, and created a lab domain that lives entirely inside the virtual environment. A Directory Services Restore Mode password was set to support recovery operations if needed.

  • Selected the option to add a new forest and specified the lab domain name.
  • Confirmed domain controller capabilities and DNS integration.
  • Reviewed database, log, and SYSVOL paths for the directory.
  • Completed the wizard and allowed the server to restart as a domain controller.

The configuration view below documents the new forest setup. It marks the point where DC01 transitions from a simple member server into the central identity provider for the lab network.

4. User lifecycle and group management in ADUC

Practiced the daily tasks a helpdesk or IAM analyst uses to manage accounts.

With the domain controller online, I opened Active Directory Users and Computers and enabled advanced features to expose more attributes and containers. In this console I created a test user, granted and removed group membership, reset passwords, and unlocked the account to simulate common support tickets.

  • Created a user named Test User with a logon name of test.user.
  • Required the user to change password at next sign in to model an onboarding flow.
  • Logged in as the new user to complete the first sign in and confirm account function.
  • Created a security group named AI and added then removed the user as a member.
  • Reset the account password and simulated an unlock after repeated sign in failures.

5. Organizational units, nested groups, and shared resources

Organized directory objects and connected group membership to a real shared folder.

To model a more realistic directory, I created a dedicated organizational unit named Custom Groups and moved the AI group into it. I then nested the AI group into a broader group and tied membership to a shared folder on the server. This demonstrates how clean structure and group based access control make permissions easier to manage over time.

  • Created an organizational unit named Custom Groups with protection from accidental deletion.
  • Moved the AI group into the new OU to separate lab objects from built in containers.
  • Nested AI into a wider group to simulate role based access for a team.
  • Created a shared folder named corp_shared and granted access through domain groups.
  • Verified that membership changes in AD controlled access to the shared network path.

Sample administrative commands

Although most of the lab is completed through Server Manager and ADUC, I also used PowerShell to interact with directory objects. These commands show how user and group management tasks can be scripted and automated once the basic workflow is understood.

Create and enable a new user 

New-ADUser -Name "Test User" `
           -GivenName "Test" `
           -Surname "User" `
           -SamAccountName "test.user" `
           -UserPrincipalName "test.user@lab.local" `
           -AccountPassword (Read-Host -AsSecureString "Password") `
           -ChangePasswordAtLogon $true `
           -Enabled $true

Group membership and password reset 

# Add user to AI group
Add-ADGroupMember -Identity "AI" -Members "test.user"

# Reset password and require change at next sign in
Set-ADAccountPassword -Identity "test.user" `
                      -Reset `
                      -NewPassword (Read-Host -AsSecureString "NewPassword")
Set-ADUser -Identity "test.user" -ChangePasswordAtLogon $true

Architecture overview

Components

  • User workstation — Client that signs in to the lab domain and accesses resources.
  • Domain controller DC01 — Windows Server 2022 host that runs AD DS and DNS.
  • Active Directory database — Stores users, groups, computers, and organizational units.
  • DNS service — Resolves host names and supports domain operations for the lab.
  • Shared folder corp_shared — File share secured by domain groups and used to test access control.

Identity and access flow

  • User signs in at the workstation with credentials linked to Test User in Active Directory.
  • Domain controller validates the password and group membership for the account.
  • Group membership places the user in AI and other lab groups that carry permissions.
  • When the user connects to \\DC01\corp_shared, the server checks these groups.
  • Changes to membership, password, or lockout state in AD immediately change access to the share.

Key insights

  • Active Directory is not only a directory service; it is the central source of truth for users, groups, and access across a Windows environment.
  • Most day to day tickets for junior analysts revolve around a few core tasks: creating accounts, resetting passwords, unlocking users, and adjusting group membership.
  • Organizational units and clean group design make permissions easier to reason about and reduce the chance of accidental access.
  • Using groups to manage access to shared folders scales much better than assigning rights directly to individual accounts.
